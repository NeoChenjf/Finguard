# 经验规则与最佳实践

本节列出开发与协作中总结的经验规则，供 agent 在日常工作中遵循，帮助避免常见错误并提高可靠性。

1. 后端服务与测试分离
- 启动服务与接口测试应分开进行，避免在测试中反复启动/关闭服务导致不稳定。
- 自动化测试优先使用 TestClient 或类似的测试客户端，而非直接启动完整服务。

2. API Key / 认证问题
- 变更密钥后务必重启服务以使新密钥生效。
- 遇到 401/502 错误时，优先排查密钥有效性、额度与配置；记录请求示例与响应供排查使用（响应中脱敏敏感信息）。

3. 前端聊天区消息管理
- 聊天区仅展示最近 5 条消息，避免本地做持久化存储。
- 页面刷新后聊天区应为空，业务逻辑需与后端会话管理分离。

4. 小程序端“[object Object]”弹窗问题
- Toast 仅显示字符串；API 错误返回应包含 `message` 字段以便展示。
- `baseUrl` 必须可在小程序环境下访问（如局域网 IP 或通过代理）。

5. 日志与调试
- 避免直接将对象转为字符串输出，优先使用 JSON 序列化并脱敏敏感字段。
- 日志应包含上下文信息（请求 ID、时间、severity），便于追溯。

6. 终端命令安全
- 禁止自动执行高风险命令（远程下载、系统级操作等）。
- 对于重要命令，先说明影响并记录执行结果与输出摘要。

7. AI 角色身份持续性
- 面向用户的 AI 助手必须在 system prompt 中明确身份，并在每次 LLM 请求注入该身份设定以保证一致性。
- 若助手忘记身份或自称模型/品牌，优先检查 system prompt 是否被覆盖或丢失。

8. 文档与产品定位
- 默认输出文档面向真实用户场景，避免面试/简历语气；如需面试/宣传版本，须用户明确说明。

9. 交接文档规范
- 每次交接需生成/更新 `next_plan.md`，写清当前状态、阶段目标、任务清单、验收标准、未决问题与参考文件。
- `next_plan.md` 应确保“下一位 agent 读完即可执行”。

10. 阶段文档规范
- 当用户要求进入或推进阶段时，必须在项目根目录创建或更新对应 `phase{N}.md`，包含三部分：阶段总览、Todo List（已完成/未完成）、阶段工作日志（含问题与解决过程）。
- 阶段过程中实时补充日志与 Todo 状态变更。

11. 阶段验收与记录
- 每次阶段任务完成后，必须执行一次验收检查（自动化优先），并将验收过程与结果写入 `phase{N}.md`。

12. 安装步骤记录
- 任何依赖/工具安装需在对应 `phase{N}.md` 的日志详情部分记录命令、结果与失败原因（如有），并写成教学性步骤以便复现。

13. 教学文档标准
- 当用户要求“教学/讲解/学习路线/教程”时，生成独立 `.md` 教学文档，结构包括：标题与目标、学习路线/分层、核心概念解释（含类比）、关键流程/步骤、真实代码片段、小结 + 自测题、下一步建议。
- 表述面向小白，短句少术语；出现术语需附白话解释；中文文档保存为 UTF-8（带 BOM）。

工作日志与其他约定
- 工作日志格式为三部分：阶段总览、Todo List、工作日志（含问题与解决、验收）。
- 关键任务完成后，应在 `next_plan.md` 或相关文档中补充变更记录。
