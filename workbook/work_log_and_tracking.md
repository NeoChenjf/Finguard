# 阶段文档、工作日志与追踪

本文件规定阶段文档（phase{N}.md）的结构与流程、工作日志格式、交接规范与任务追踪方式。

---

## 1. 阶段文档四章结构

当用户要求进入或推进某个阶段时，必须在项目根目录创建或更新 `phase{N}.md`，统一采用以下四章结构：

```
# Phase N — 阶段标题

## Chapter 1：Spec 前置计划（冻结）
说明：汇总本阶段关键 Spec 的最终决策与推荐理由，作为统一基线。
### 1) Spec 条目标题
- 决策/推荐：具体方案
- 理由：为什么选择该方案
- 详情：参数、边界、约束（按需）
（每个 Spec 条目编号递增）

## Chapter 2：Phase N 计划
完成步骤：
1. （编号步骤列表，每步简述做什么）
验收项目：
1. （编号验收项，每项描述验收条件与 Pass/Fail 标准）

## Chapter 3：任务完成详情
（阶段结束后补写）
### 步骤执行记录
- 步骤 N：xxx —— 状态（已完成/进行中/跳过）
### 问题与解决
- 问题 N：现象 → 定位 → 处理
（若无问题标注"未遇到"）

## Chapter 4：验收详情
（阶段结束后补写）
### 验收清单
| # | 验收项 | 状态 | 说明 |
### 产物列表
| 文件 | 说明 |
### 变更日志
- [YYYY-MM-DD] 变更说明
```

### 操作流程

1. **阶段启动时**：先写 Chapter 1（Spec）+ Chapter 2（计划），讨论确认后冻结。
2. **阶段执行中**：实时在 Chapter 3 记录执行与问题。
3. **阶段结束时**：补齐 Chapter 4 验收详情与产物清单。

### 命名约定

- 标题格式：`# Phase N — 阶段名称`
- 章节格式：`## Chapter N：章节标题`
- Spec 条目：用 `### N)` 编号

## 2. Spec 前置收敛流程

进入某阶段前，必须先一次性收敛该阶段的所有关键 Spec（参数、策略、验收口径），再集中实现。避免边做边改导致返工。

操作步骤：
1. 列出该阶段必须确定的 Spec 清单。
2. 逐项讨论确认，结论写入 `phase{N}.md` Chapter 1。
3. Spec 完整冻结后再启动实现（Chapter 2 步骤）。

## 3. 阶段验收与记录

- 每个阶段完成后，必须执行验收检查（自动化优先），将过程与结果写入 Chapter 4。
- 验收工具不可用时，允许使用简化替代方案，但必须在验收文档中写明"替代方案 + 限制 + 改进建议"。

## 4. 问题记录规范

执行阶段计划时，必须在 Chapter 3 中记录每一步遇到的问题/报错与解决方案。若未遇到问题，明确标注"未遇到问题"。

## 5. 工作日志格式

每次工作日志包含：
1. 阶段总览（本阶段目标概述）
2. Todo List（已完成 / 未完成）
3. 执行日志（问题与解决、重要命令与结果、验收记录）

### 日志顺序与补录

- 所有日志按时间顺序排列。
- 新内容默认追加到时间线末尾；补录历史记录须插入正确时间位置并标注日期。
- 同一日期内多次记录按执行时间从早到晚排序。

## 6. 交接与 next_plan.md

每次交接需生成或更新 `next_plan.md`，内容包括：
- 当前状态
- 阶段目标
- 任务清单
- 验收标准
- 未决问题与参考文件

目标：**下一位 agent 读完即可执行**。

## 7. 任务追踪

- 使用 `manage_todo_list` 跟踪多步骤任务，确保目标、完成条件、当前状态清晰。
- 重要变更时记录历史。

## 8. 安装与环境记录

任何依赖/工具安装步骤必须写入对应 `phase{N}.md` Chapter 3，包含命令、执行结果与失败原因（如有），写成可复现的步骤。

## 9. 单阶段文档唯一事实源（禁止并行日志）

- **一件事只保留一个阶段文档**：同一事项的执行、问题、验收、产物，必须全部回填到对应 `phase{N}.md`（或该事项唯一指定的阶段文档）中。
- **禁止并行记录同类状态**：不再额外创建“完成日志/临时总结/进度快照”这类与阶段文档重复的信息文件（除非用户明确要求单独产物）。
- **中间成果回填原则**：若执行中临时创建了辅助记录文件，阶段结束前必须将有效内容合并回阶段文档，并删除或归档辅助文件，避免双份事实源。
- **交接读取优先级**：交接时只以阶段文档为准；`next_plan.md` 只做“下一步指引”，不承载与阶段文档重复的详细执行事实。
- **验收口径一致性**：Chapter 4 的验收清单、产物列表、变更日志必须与仓库实际变更一致，禁止在其他文档维护另一套验收状态。
