# 工作日志、阶段记录与追踪

## 工作日志格式
- 每次工作日志包含三部分：
	1) 阶段总览（本阶段目标概述）
	2) Todo List（已完成 / 未完成，含负责人/预计完成时间）
	3) 工作日志（问题与解决、重要命令与结果、验收记录）

## 日志顺序与补录规则
- 所有“验收结果 / 执行日志 / 诊断记录”必须按时间顺序排列。
- 新内容默认追加到时间线末尾；若补录历史记录，必须插入正确的时间位置，并在条目内标注日期。
- 同一日期下的多次记录，按执行时间从早到晚排序；若时间不明，按补录顺序但需说明“不确定具体时间”。

## 阶段与验收
- 当用户提出“开始阶段任务/推进/进入某阶段”时，必须在项目根目录创建或更新 `phase{N}.md`：
	- 三部分固定：阶段总览、Todo List、阶段工作日志（含问题与解决过程）。
	- 阶段完成后执行验收检查（优先自动化），并将验收结果写入 `phase{N}.md` 的验收部分。

## 交接与下一步计划
- 每次交接需生成/更新 `next_plan.md`，内容包括当前状态、阶段目标、任务清单、验收标准、未决问题与参考文件，确保“下一位 agent 读完即可执行”。

## 任务追踪
- 使用 `manage_todo_list` 跟踪多阶段任务，确保目标、完成条件、当前状态清晰，并在重要变更时记录历史。

## 安装与环境记录
- 任何依赖/工具安装步骤必须写入阶段日志，包含命令、执行结果与失败原因（如有），并且写成教学性步骤以便复现。

## 示例条目
- Todo 示例：
	- [ ] 安装 venv（负责人：张三，预计：2026-02-01）
	- [x] 更新接口文档（负责人：李四，完成：2026-01-27）
