# 腾讯云数据库团队面试总结 (2026-03-12)

## 1. 面试概况
*   **岗位方向**：云数据库后端/内核开发
*   **面试核心**：项目深度、OS/网络基础、云数据库架构理解、算法实操
*   **项目背景**：Thing Guard (Asset Allocation with LLM)

## 2. Pros & Cons (优势与不足)

### ✅ Pros (亮点)
1.  **工程实践闭环**：能够清晰描述从需求分析（用户画像）到系统实现（10阶段规划）的全过程。
2.  **性能优化意识**：在压测中发现 YAML 磁盘读取瓶颈，并通过引入 5s 缓存机制将延迟从 640ms 降至 200ms，展示了对 P99/P95 指标的敏感度。
3.  **技术应变能力**：面对 SSL 证书不匹配导致的方案失败，能迅速降级为 `curl` 方案确保业务可用。
4.  **跨学科潜力**：保研背景，具备良好的结构化思维，能将复杂的系统问题类比（如毕加索、货架商品等）进行解释。

### ❌ Cons (痛点)
1.  **底层原理深度稍显薄弱**：在被深挖 `fork()` 子进程 FD 继承细节、文件监控（写入即显示）的动态实现以及大文件处理（内存映射）的边界情况时，回答显得不够透彻，存在“用实践经验填补理论空白”的痕迹。
2.  **代码实操与健壮性**：算法题（原数组去重）在 ACM 模式下表现略显生疏，针对特殊用例（如 0.0.9 或极长数据）的处理逻辑不够优雅，暴露出对静态初始化与动态结果处理的经验欠缺。
3.  **工业认知鸿沟**：对云数据库“货架化”架构和分布式 CAP 理论的权衡（CP 模式）认识处于理论层面，缺乏对数据库内核设计模式（专家架构 vs 组合架构）的直观理解。

## 3. 待巩固知识点清单 (Weak Points)

### 操作系统 (OS)
*   **进程模型深度**：理解 `fork()` 的本质是“写时复制” (COW) 产生的新进程。掌握子进程对父进程文件描述符 (FD) 的**描述符复制**（FD 值相同）与**内核对象共享**（共享文件偏移量和状态标志）。理清 `vfork`（挂起父进程、共享地址空间）与 `clone`（高度可配置的创建方式，`pthread_create` 的底层）的差异。
*   **高性能 IO 处理**：`mmap` 的边界情况与性能权衡、`tail` 命令的底层高效实现原理。
*   **实时监控机制**：`inotify` / `kqueue` 机制实现“写入即显示”，以及由于延迟写入导致的监控盲点处理。
*   **并发编程**：指针追踪文件时的竞态条件 (Race Condition)、死锁防范、内存屏障 (Memory Barrier) 与原子操作。

### 网络与通信 (Network)
*   **SSL/TLS 握手详情**：证书不匹配的深层原因及更可靠的修复/降级方案，而非简单的调用 `curl`。
*   **HTTP 链路效率**：分析 API 请求耗时在网络各层的分布，识别瓶颈。

### 分布式系统与数据库 (DB)
*   **一致性协议**：`Raft` 或 `Paxos` 协议的细节、云数据库在分布式场景下的 `CP` 权衡。
*   **数据库存储架构**：`B+ 树` vs `LSM 树`、`Buffer Pool` 的替换策略、工业级缓存与“5s 缓存”的差距。
*   **架构设计模式**：`专家架构` (Top-down) vs `组合架构` (Bottom-up) 在大型系统中的应用场景。

### 算法与工程 (Algorithms & Coding)
*   **ACM 模式**：标准输入输出处理、处理大数据量时的内存限制。
*   **健壮性处理**：对边界值（空值、浮点精度如 `0.0.9`、极长数组）的优雅处理。
*   **空间复杂度**：在原数组上进行操作的指针技巧。
