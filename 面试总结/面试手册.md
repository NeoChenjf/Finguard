# FinGuard 项目面试手册

## 文档说明

本文档记录基于 FinGuard 项目的面试问答，用于准备技术面试和项目答辩。

---

## 问题 1：为什么选择用 C++ 来实现这个资产配置决策支持系统，而不是 Python 或其他语言?

### 回答

我选 C++，说白了就三点：

1. 这个系统要扛得住量。多人同时来问、还要边生成边返回，后端不能卡。
2. 这个系统要稳。流量大了要拦一拦，下游挂了要自动保护，超时了要及时收手，不能把整条链路拖死。
3. 这个系统要可控。给用户的结果不能乱来，要能追溯、能解释、能回看。

Python 我也在用，但主要放在数据处理和脚本工具上；核心服务层我用 C++，因为它在“速度、稳定、可控”这三件事上更合适。

一句话总结：我不是为了秀技术选 C++，是为了让系统在真实流量下又快又稳。

## 问题 2：从问题1引申出的核心概念树，以及各概念的核心意义

### 2.1 树状思维导图（大白话）

问题1：为什么选 C++
├─ 先把系统做快
│  ├─ 用户发请求别排太久
│  └─ 回复可以边生成边看到
├─ 再把系统做稳
│  ├─ 人太多就限流
│  ├─ 下游挂了就暂时断开
│  ├─ 等太久就超时
│  └─ 失败了有备用方案
├─ 再把结果做可控
│  ├─ 输出有规则
│  ├─ 出问题能查日志
│  └─ 指标能看健康状态
└─ 最后用压测和剖析去找瓶颈
   ├─ 先用假下游测自己
   └─ 再接真下游测全链路

### 2.2 面试追问怎么解释（简版）

- **SSE**：不是等全部生成完才返回，而是边生成边发给前端，用户体感更快。
- **风控中台**：把规则放在后端统一管，不让每个前端自己乱写一套。
- **可控输出**：系统不只要“能回答”，还要“回答得稳、可查、可解释”。
- **限流**：人太多就排队或拒绝一部分，先保系统不崩。
- **熔断**：下游连续出错就先别打它，等它恢复了再试。
- **超时/重试/降级**：等太久就停，能重试就重试，还不行就走备用结果。
- **日志和指标**：日志看细节，指标看趋势，两个一起才好排障。
- **Mock 下游**：先用假下游测自己代码有没有瓶颈，再看真实外部服务的影响。

### 2.3 30 秒收口话术（大白话）

这套系统说到底就是三步：先跑得快，再跑得稳，最后结果要可控可追溯。
我选 C++ 是因为它更适合把这三件事一起做好，尤其在高并发场景下更稳当。

## 问题 3：你在项目里模板编程的具体场景？如何支持不同风险等级策略？

### 回答（大白话版）

#### A. 现在项目里模板具体用在哪

我把“配置缓存”这件事做成了一套通用模板。
简单说就是：
- 不管是限流配置、超时配置、熔断配置，还是并发配置
- 都走同一个缓存逻辑（过期再刷新、并发安全、减少重复读文件）

这样我不用给每种配置都抄一遍差不多的代码。

#### B. 风险等级策略目前怎么做

风险等级（小白、经验型、专业型）现在是走配置文件，不是模板。
也就是运行时按用户等级读对应规则：
- 小白更保守
- 经验型居中
- 专业型更放开

这个做法的好处是改规则快，不用改代码重新发版。

#### C. 为什么这样设计

- **模板放在通用基础层**：把重复活（缓存、刷新、并发保护）统一起来。
- **业务规则放在配置层**：让产品和风控同学能快速调参数。

一句话：模板负责“省代码、提稳定”，配置负责“好调整、好运营”。

## 问题 4：限流/熔断/并发控制里，我是怎么用模板做复用的？

### 回答（先讲真实实现，再讲可扩展模板）

#### 1) 真实实现：模板复用在“配置缓存层”，不在算法本体

我当前项目里：
- `TokenBucket`（限流）是独立类
- `CircuitBreaker`（熔断）是独立类
- `ConcurrencyLimiter`（并发控制）是独立类

它们没有直接写成模板类。真正用模板复用的地方是 `reliability_config.cpp` 里的配置缓存器：
- `template <typename T> struct ConfigCache`
- `template <typename Loader> T get(Loader loader)`

这层统一服务 `RateLimitConfig / TimeoutConfig / CircuitBreakerConfig / ConcurrencyConfig`，把“热路径避免频繁读文件”的逻辑复用了。

#### 2) 我设计的“通用模板”是什么

这个通用模板的职责不是替代限流/熔断算法，而是提供统一配置生命周期管理：
- TTL 过期判断
- 惰性刷新
- 线程安全访问
- 返回当前可用配置

也就是：**算法各自独立，配置缓存统一复用**。

#### 3) 模板参数化部分是什么

已落地的参数化有两层：
- `T`：配置类型（`RateLimitConfig`、`TimeoutConfig`、`CircuitBreakerConfig`、`ConcurrencyConfig`）
- `Loader`：加载函数（例如 `load_rate_limit_config`）

可扩展时还可以继续参数化：
- `ClockPolicy`：时间源策略（真实时钟/测试时钟）
- `LockPolicy`：并发策略（`mutex` / `shared_mutex`）
- `ValidationPolicy`：配置合法性校验策略

#### 4) 相比写多个不同类的优势

- **少重复**：不用为四种配置各写一套缓存与过期逻辑
- **一致性**：刷新时机、并发保护、行为语义保持一致
- **性能稳定**：编译期类型绑定，无虚函数分发开销
- **可维护**：修一个缓存逻辑，四类配置一起受益

#### 5) 面试官追问“为什么不把限流器/熔断器也模板化？”

我的回答是：
限流、熔断、并发控制的状态机和行为差异很大，强行做一个通用算法模板会增加认知负担。
当前更优取舍是：
- 算法层保持独立类（清晰、可读）
- 基础设施层（配置缓存）做模板复用（高复用、低风险）

### 一句话收口

我没有把三种治理算法硬抽成一个模板，而是把它们共同需要的“配置缓存与刷新机制”模板化，这样复用充分、边界清晰、性能也稳定。

## 问题 5：能不能具体讲讲你是怎么设计这个"缓存模板"的？

### 回答（大白话版）

#### 1) 模板的参数化部分是什么？

简单说就是"填空题"，有三个空：

- **T（配置类型）**：这是什么配置？限流配置？超时配置？熔断配置？
- **Loader（怎么加载）**：从哪儿读？读文件？查数据库？调 API？
- **Validator（怎么验证，可选）**：读完后检查合法性，比如"QPS 不能是负数"

打个比方：
- T 就像"这是什么菜"（青菜、肉、鱼）
- Loader 就像"怎么做"（炒、蒸、煮）
- Validator 就像"尝味道"（太咸了要调）

#### 2) 你是如何保证线程安全的？

用了**互斥锁（mutex）**，像银行柜台排队：

- **读配置时（多人可以一起读）**：
  - 如果配置没过期，大家直接拿走，不用等（快速路径）
  - 如果要刷新，加独占锁，一个人刷新、其他人等一下

- **刷新配置时（只能一个人改）**：
  - 加锁 → 重新加载 → 验证 → 替换旧配置 → 解锁
  - 保证别人要么看到旧的完整版，要么看到新的完整版，不会看到改到一半的

具体场景：
- **场景1**：100 个请求同时来读限流配置，配置还没过期 → 不用锁，直接返回（快）
- **场景2**：配置过期了，第 1 个请求发现了 → 加锁刷新，其他 99 个等一下 → 刷新完大家一起用新的

#### 3) 这种设计相比写多个不同缓存类的优势？

**从代码量看：**
- 不用模板：限流缓存一套代码、超时缓存一套、熔断缓存一套、并发缓存一套 = 4 套
- 用模板：1 套代码，换 4 个参数 = 1 套

**从维护看：**
- 不用模板：发现缓存有 bug，要改 4 个地方，容易漏
- 用模板：改 1 个地方，4 个一起好

**从一致性看：**
- 不用模板：4 套代码各写各的，过期时间可能不统一、刷新逻辑可能不一样
- 用模板：行为完全一致，不会出现"限流缓存 5 秒刷新、超时缓存 10 秒刷新"这种不一致

**从性能看：**
- 模板在编译时就确定了类型，不是运行时动态判断，速度和手写一样快

### 一句话收口

就像搭积木，我做了一个"缓存积木模板"，换不同参数就能拼出限流缓存、超时缓存、熔断缓存，写一次代码、用四次，改起来也方便。
