# 令牌桶限流基础

## 标题与目标
理解令牌桶如何限制请求速率，并能解释“突发流量”为什么允许。

## 学习路线 / 分层
1. 先理解“限流”是让系统不过载。
2. 再理解令牌桶的“令牌产生 + 消耗”。
3. 最后理解参数：速率、桶容量、突发。

## 核心概念解释（含生活类比）
- 令牌桶：门口的“取号机”，每秒放出固定数量的号。
- 速率：每秒放出的号数。
- 桶容量：最多能存多少号，决定可承受的突发请求。

## 关键流程 / 步骤
1. 系统按固定速率生成令牌（上限为桶容量）。
2. 每个请求来时先拿令牌，拿到则通过。
3. 没有令牌则拒绝或排队。
4. 可按用户/接口/模型设置不同桶。

### 关键流程步骤详情（补充知识点）
- 令牌是“准入许可”，不是排队序号；拿到令牌后默认立刻执行，不保证顺序。
- 是否“拒绝”或“排队等待”取决于系统策略：令牌桶只决定能否通过，不负责排队。
- 同一时刻多个请求各自拿到令牌，可并发执行；并发上限通常由线程池/连接数等其他组件决定。
- “按用户/接口/模型”本质是用不同 key 区分桶：
  - 用户：`bucket:user:<user_id/api_key>`
  - 接口：`bucket:route:</api/v1/...>`
  - 模型：`bucket:model:<model_name>`
  - 也可组合维度（如 `user + route`、`user + model`）以更细粒度限流。

## 真实代码片段（C++ 伪代码，易改造）
```cpp
struct TokenBucket {
  double rate;   // 每秒生成多少令牌
  double cap;    // 最大令牌数
  double tokens; // 当前令牌
  std::chrono::steady_clock::time_point last;

  bool allow() {
    auto now = std::chrono::steady_clock::now();
    double secs = std::chrono::duration<double>(now - last).count();
    tokens = std::min(cap, tokens + secs * rate);
    last = now;
    if (tokens >= 1.0) { tokens -= 1.0; return true; }
    return false;
  }
};
```

## 小结 + 自测问题
小结：令牌桶能稳定速率，同时允许短时突发。

自测：
1. 为什么桶容量越大突发越强？
2. 如果 rate=10/s、cap=20，突发最多能放行多少请求？

自测题答案：
1. 桶容量越大可存的令牌越多，瞬间可用令牌越多，允许更大突发。
2. 最多可立即放行 20 个请求（桶满时）。
