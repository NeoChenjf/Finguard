# 缓存补充 2：位图、哈希函数与热点 key

## 标题与目标
理解位图、哈希函数、热点 key 与热点 key 过期的基础概念，并能用通俗语言解释它们的作用。

## 学习路线 / 分层
1. 先理解位图是什么（0/1 小格子）。
2. 再理解哈希函数如何把 key 映射到位图格子。
3. 最后理解热点 key 与热点 key 过期带来的风险。

## 核心概念解释（含生活类比）
- 位图：一排开关，开/关表示 1/0；像仓库里一排灯，亮=存在，灭=不存在。
- 哈希函数：把“物品名称”转换成“开关编号”的规则；像给每个物品分配若干个灯位。
- 热点 key：被很多人频繁访问的同一个物品；像热门饮料被反复点单。
- 热点 key 过期：热门饮料的标签突然失效，大家同时去仓库补货。
### 详情说明
- 位图本质就是一排 0/1 序列，每一位表示“被标记/出现过”。在布隆过滤器里，它不是直接存“这个物品是否存在”，而是存“哈希映射到这里的 key 曾经出现过”。  
- 哈希函数可以理解为把“物品名字”和“开关编号”联系起来的规则；不同物品可能映射到同一组开关，所以只保证“可能存在”。  
- 布隆过滤器的使用方式：如果判断“明显不存在”，就直接不再查；如果判断“可能存在”，再去缓存或源头做真实确认。  
- 想确切知道是否真的存在，必须访问真实存储（缓存或源头数据库/文件/服务），布隆过滤器无法 100% 确认。  
- “热点 key 过期”里的“仓库”指源头（数据库/文件/服务）；过期后大家同时回源补货，压力骤增。

## 关键流程 / 步骤
1. 位图存储：用 0/1 表示“是否被标记”。
2. 哈希映射：把 key 变成若干个索引位置。
3. 插入：把这些位置都置为 1。
4. 查询：检查这些位置是否全为 1。
5. 热点识别：统计访问次数，频率最高的 key 即为热点 key。
6. 热点过期风险：热点 key 过期时，大量请求会同时打到源头。
### 详情说明
- “插入”发生在把一个 key 加入布隆过滤器时，常见场景是源头确认存在后加入，或系统初始化批量加载已有 key。  
- “查询检查多个位置”指的是对同一个 key 检查多个位（多个哈希映射的位置），不是同时查多个 key。  
- 位图里的“位”就是 0/1 的小格子；一个 key 会映射到多个位，而一个位也可能被多个 key 共享。  
- 布隆过滤器判断“可能存在”后，仍需到缓存/源头做真实确认；“明显不存在”才直接拒绝。  

## 真实代码片段（伪代码示例）
```cpp
// 伪代码：位图 + 哈希映射的最简示意
const int N = 1000000; // 位图长度
bitset<N> bitmap; // 位图，用 0/1 表示

int h1(const string& k) { return hash1(k) % N; } // 哈希函数 1
int h2(const string& k) { return hash2(k) % N; } // 哈希函数 2

void add_key(const string& k) { // 插入 key
  bitmap[h1(k)] = 1; // 置位 1
  bitmap[h2(k)] = 1; // 置位 1
}

bool may_exist(const string& k) { // 查询是否可能存在
  return bitmap[h1(k)] && bitmap[h2(k)]; // 两个位置都为 1 才可能存在
}
```

## 小结 + 自测问题
小结：位图省空间，哈希把 key 映射到位图；热点 key 过期会触发大量回源。

自测：
1. 为什么布隆过滤器会“可能误判为存在”？
2. 热点 key 过期会导致什么问题？

自测题答案：
1. 不同 key 可能把相同位图位置置为 1，导致查询时“全为 1”但实际不存在（假阳性）。
2. 大量请求同时回源刷新，造成源头压力骤增和延迟上升。

- 学习热点检测与预热策略。
