# 缓存补充 1：穿透/击穿/雪崩 与容量/TTL

## 标题与目标
理解缓存穿透/击穿/雪崩的基础概念，并能用简单方法评估缓存容量与 TTL。

## 学习路线 / 分层
1. 先区分“穿透、击穿、雪崩”的触发场景。
2. 再理解最常用的三类防护手段。
3. 最后学会估算容量与 TTL 的思路。

## 核心概念解释（含生活类比）
- 穿透：找不存在的物品，抽屉里没有，只能反复去仓库问。
- 击穿：大家同时来找一个热门物品，抽屉突然空了。这里的“空”指的是这个热门物品对应的位置空了（比如过期或被拿走刷新），不是整抽屉没有别的物品；因为大家都在找同一个物品，所以会同时去仓库请求补货，导致压力瞬间增大。
- 雪崩：很多抽屉同时清空，所有人都涌向仓库。
- 关键区分：
  - 空结果缓存（防穿透）只用于“物品本来就不存在”的情况，作用是挡住无效请求；因此你不会得到该物品，因为它本来就没有。
  - “别人正在用/刷新”属于击穿场景，正确做法是让一个人去仓库取，其它人等待或拿旧值，而不是缓存空结果。

## 关键流程 / 步骤
1. 穿透场景：请求的 key 不存在，缓存和源头都没有，造成无效访问。
2. 击穿场景：某个热点 key 过期，瞬间大量请求打到源头。
3. 雪崩场景：大量 key 同时过期，源头被压垮。
4. 防护思路（逐项解释）：
   - 穿透-空值缓存：当源头查不到数据时，缓存一个“空结果”并设置较短 TTL；后续相同请求直接命中空值，避免反复打到源头。
   - 穿透-布隆过滤器：在请求进入缓存前先用布隆过滤器判断“是否可能存在”；若明确不存在则直接拒绝，减少无效访问。布隆过滤器用“位图 + 多个哈希函数”来判断：位图是一串 0/1 的小格子（像一排开关），哈希函数把 key 映射到若干个格子上并把它们置 1；查询时检查这些格子是否全为 1。特点是“可能误判为存在，但不会把存在的误判为不存在”。
   - 击穿-互斥锁：热点 key 过期时只允许一个请求去源头刷新，其它请求等待或短暂返回旧值，避免并发把源头压垮。
   - 击穿-热点预热：在热点 key 快过期前提前刷新或定时刷新，尽量不让它在高峰期过期。
   - 雪崩-过期时间随机化：给一批 key 的 TTL 加上随机抖动，避免同时过期。
   - 雪崩-分批刷新：把大批量 key 分时段刷新或按优先级刷新，降低瞬时压力。
5. 容量评估：按“热点数据量 + 访问分布”估算，优先满足高频数据。
6. TTL 评估：平衡“新鲜度”和“源头压力”，热点短 TTL、非热点长 TTL。

## 真实代码片段（伪代码示例）
```cpp
// 伪代码：请求流程中加入空值缓存与随机 TTL
if (cache.has(key)) { return cache.get(key); } // 命中缓存直接返回
if (!db.exists(key)) { // 源头不存在该数据
  cache.set(key, NULL_VALUE, ttl_short); // 缓存空值，避免穿透
  return NOT_FOUND; // 返回未找到
}
value = db.get(key); // 从源头读取
cache.set(key, value, ttl_base + rand(0, ttl_jitter)); // 写入缓存并加入随机 TTL
return value; // 返回结果
```

## 小结 + 自测问题
小结：穿透/击穿/雪崩是三种典型缓存风险，容量和 TTL 决定了“命中率与压力”。

自测：
1. 为什么雪崩比击穿更危险？
2. TTL 设置太短会带来什么问题？

自测题答案：
1. 雪崩是大量 key 同时过期，影响面更大、并发回源更多，更容易把源头压垮。
2. 过短会导致频繁过期和回源，命中率下降、源头压力上升。
