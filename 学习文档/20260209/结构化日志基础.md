# 结构化日志基础

## 标题与目标
理解“结构化日志”是什么，以及为什么比纯文本日志更好检索。

## 学习路线 / 分层
1. 先理解“日志是排障证据”。
2. 再理解结构化日志是“字段化”。
3. 最后理解如何设计字段与级别。

## 核心概念解释（含生活类比）
- 结构化日志：每条日志像一张表格记录。
- 字段：固定列，比如时间、级别、trace_id。
- 好处：可以精准检索、聚合统计。

## 关键流程 / 步骤
1. 定义日志字段（时间、级别、trace_id、模块、耗时）。
2. 输出统一 JSON 结构。
3. 关键路径记录错误码与原因。
4. 与 trace_id 结合实现链路定位。
### 详情说明
- 结构化日志强调“标准格式 + 固定字段”，例如时间、事件类型、等级、trace_id、模块、耗时等。  
- “耗时”指处理这次请求或某个步骤所花的时间。  
- “模块”指日志来自哪个功能区域/组件（谁在记录），不是请求类型。  
- 在 FinGuard 中可理解的模块示例：`server`（HTTP/SSE 路由）、`core`（主流程）、`risk`（风控规则）、`llm`（模型调用）、`data`（数据源/行情）、`util`（通用工具）。  
- 关键路径指对业务结果最重要的链路（如接口入口、主流程编排、风控闸门、模型调用、关键数据读取），需要详细记录。  
- 非关键路径指辅助性步骤（如缓存淘汰、调试采样、低频后台任务），可简化记录。  
- 关键路径建议记录：`trace_id`、耗时、错误码、错误原因、事件类型。  
- 非关键路径建议记录：事件类型 + 少量字段，日志级别可用 DEBUG/INFO。  
- 链路定位：通过同一个 `trace_id` 串起一次请求经过的所有环节，用来判断问题出在哪一步。  
#### 示例（关键路径）
```json
{"ts":"2026-02-09T10:15:30Z","level":"INFO","trace_id":"e7f3c7b9-7a6d-4b4f-9b86-4b5f7b9a1c2d","module":"risk","event":"rules_checked","latency_ms":12,"code":"RISK.OK","detail":"no warning"}
```
```json
{"ts":"2026-02-09T10:15:31Z","level":"ERROR","trace_id":"e7f3c7b9-7a6d-4b4f-9b86-4b5f7b9a1c2d","module":"llm","event":"call_failed","latency_ms":3000,"code":"LLM.TIMEOUT","reason":"upstream timeout"}
```
#### 示例（非关键路径）
```json
{"ts":"2026-02-09T10:15:32Z","level":"DEBUG","module":"util.cache","event":"evict","key":"profile:123"}
```

## 小结 + 自测问题
小结：结构化日志让你更快定位问题，也便于做指标统计。

自测：
1. 为什么统一字段比自由文本更易分析？
2. trace_id 在结构化日志里起什么作用？

自测题答案：
1. 统一字段便于索引、过滤和聚合统计，自动化分析更容易。
2. 用于把同一请求的日志串起来，便于定位问题。
