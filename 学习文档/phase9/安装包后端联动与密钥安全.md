# 安装包如何带着后端跑起来：Sidecar + Config + API Key 安全实战（Phase 9）

> 目标：看完后你能讲清楚三件事：安装包为什么自带后端能力、首次启动“复制 config”到底复制了什么、如何避免 API Key 随安装包泄露。

## 1) 学习路线（从易到难）

1. 先理解“两个 exe + 多个 DLL”各自负责什么。
2. 再理解安装后第一次启动时，Rust 侧如何复制 `config/` 到 `%APPDATA%/finguard/`。
3. 再理解前端是如何连到后端 sidecar 的（`127.0.0.1:8080`）。
4. 最后做安全收口：新电脑如何可用且不泄露你的默认 Key。

## 2) 核心概念解释（白话 + 类比）

### 2.1 安装包里的“完整功能”是怎么来的？
- 不是“到用户电脑再编译后端”，而是你在构建时就把后端产物一起打进安装包。
- 你的包里有：
  - 桌面主程序：`finguard-desktop.exe`（Tauri 壳）
  - 后端 sidecar：`finguard.exe`
  - 后端依赖 DLL：`drogon.dll`、`libssl-3-x64.dll` 等
  - 配置模板：`config/llm.json` 等
- 安装后点击桌面程序，它会自动拉起 sidecar。

### 2.2 “通过 DLL 达到后端功能”这句话哪里对、哪里不对？
- 对的一面：没有 DLL，`finguard.exe` 可能启动失败（缺依赖）。
- 更准确的说法：
  - **业务功能主体**在 `finguard.exe`（你编译出的后端程序）
  - **DLL 主要是运行时依赖库**（网络、TLS、框架等）
- 所以不是“只有 DLL 才有功能”，而是“exe + DLL 一起构成可运行后端”。

### 2.3 首次启动“复制 config”到底是什么动作？
- 复制源：安装目录下的 `config/`
- 复制目标：`%APPDATA%/finguard/config/`
- 触发条件：仅当 `%APPDATA%/finguard/config/llm.json` 不存在时才复制
- 目的：
  - 避开 `Program Files` 的写权限限制
  - 让后端继续按 `cwd/config/llm.json` 读写

### 2.4 前端如何连到后端？
- 桌面前端请求 `http://127.0.0.1:8080`。
- 这个端口由 sidecar `finguard.exe` 提供 API。
- 流程是：`finguard-desktop.exe` 启动 -> 拉起 `finguard.exe` -> 前端请求本地 API。

### 生活类比（电影院）
- `finguard-desktop.exe` 像“售票大厅”。
- `finguard.exe` 像“放映机房”。
- DLL 像“机房配套设备”。
- `config/` 像“放映参数表”，首次拷到 `%APPDATA%` 后，之后都在“本地运营目录”改参数。

## 3) 关键流程/步骤（可复刻）

1. 构建阶段把 sidecar 与资源打包进安装包：
   - `externalBin` 打入 `finguard`。
   - `resources` 打入 DLL 与 `config/*`。
2. 用户安装后，目录里具备可执行与依赖文件（无需再编译 C++ 后端）。
3. 用户首次启动桌面程序：
   - 检查 `%APPDATA%/finguard/config/llm.json` 是否存在。
   - 若不存在，从安装目录复制 `config/` 到 `%APPDATA%/finguard/config/`。
   - 设置 sidecar CWD 为 `%APPDATA%/finguard/`。
   - spawn `finguard.exe`。
4. 前端请求 `127.0.0.1:8080`，由 sidecar 提供后端能力。
5. 用户在设置页改 Key/模型后，后端写回 `%APPDATA%/finguard/config/llm.json`。

## 4) 真实代码片段（逐行注释）

```rust
// frontend/src-tauri/src/lib.rs
fn ensure_config_in_appdata(install_dir: &Path) {
    let src_config = install_dir.join("config");
    let dst_config = appdata_work_dir().join("config");

    if dst_config.join("llm.json").exists() {
        return; // 已存在则不覆盖用户配置
    }

    std::fs::create_dir_all(&dst_config).ok();
    for entry in std::fs::read_dir(&src_config).unwrap() {
        let src_file = entry.unwrap().path();
        if src_file.is_file() {
            let dst_file = dst_config.join(src_file.file_name().unwrap());
            let _ = std::fs::copy(&src_file, &dst_file);
        }
    }
}
```

- 第 1 行：定义“首次复制配置”的函数。
- 第 2 行：拼出安装目录中的 `config/` 源路径。
- 第 3 行：拼出 `%APPDATA%/finguard/config/` 目标路径。
- 第 5-7 行：若目标已有 `llm.json`，直接返回，避免覆盖用户已改配置。
- 第 9 行：创建目标目录（不存在就创建）。
- 第 10 行：遍历源配置目录。
- 第 11 行：拿到每个源文件路径。
- 第 12-15 行：将每个文件复制到 AppData 的 config 目录。

## 5) 小结 + 自测题（含答案）

### 小结
- 新电脑不需要“再构建后端”，安装包已包含后端 exe 与依赖 DLL。
- 桌面程序负责拉起 sidecar，前端通过本地 `127.0.0.1:8080` 调后端。
- 首次复制 config 是“安装目录 -> `%APPDATA%/finguard/config/`”，目的是可写与可持久化。
- API Key 是否泄露，关键看你是否把真实 Key 放进了打包输入的 `config/llm.json`。

### 自测题
1. 为什么要把 config 复制到 `%APPDATA%`，而不是一直用安装目录？  
   - 答：因为安装目录（如 `Program Files`）常不可写，而后端需要读写 `config/llm.json`。
2. 新电脑安装后为什么能直接跑后端？  
   - 答：因为后端是已编译产物（`finguard.exe` + DLL）并随安装包分发。
3. API Key 会不会泄露？  
   - 答：如果你把真实 Key 打进安装包里的 `config/llm.json`，就会泄露风险；应只放占位值，让用户首次自行填写。

## 6) 下一步建议（安全发布必做）

1. 发布前把 `frontend/src-tauri/binaries/config/llm.json` 的 `api_key` 改成占位符，不要放真实 Key。
2. 若历史上打过含真实 Key 的安装包，立刻到供应商后台轮换（revoke + regenerate）该 Key。
3. 保留“用户首次在设置页输入 Key”的流程，写入 `%APPDATA%/finguard/config/llm.json`。
4. 可选增强：优先读环境变量 `FINGUARD_LLM_API_KEY`，把明文 Key 从文件配置降到最低。
