# 2026-02-01 学习总览：Phase 2 深度排查与关键概念

## 标题与目标

读完这份文档，你能说明：为什么 Drogon 直连会失败、为何 curl fallback 能跑通，以及排查网络问题时应关注哪些关键点（IPv6 / SNI / DNS / TLS）。

## 整体学习路线

1. 理解“直连 vs 降级（curl fallback）”的差异。
2. 区分“客户端->后端”与“后端->外网”两段网络路径。
3. 关注 IPv6 / c-ares / SNI 等可能导致直连失败的点。
4. 学会用工具验证与定位（curl / 日志 / 抓包）。

## 学习详情

### 1) 直连失败与降级成功

- Drogon HttpClient 直连外部 LLM 失败，curl fallback 可成功返回。
- 说明问题更可能出在“后端->外网”链路或底层网络栈，而不是业务逻辑。

### 2) IPv4 与 IPv6

- 部分环境中 IPv6 路由不可达，但 DNS/c-ares 会优先返回 IPv6。
- 结果：Drogon 直连走到“不可达 IPv6 路径”导致失败。

### 3) c-ares 与 SNI

- c-ares 可能优先 IPv6，导致错误路线被选中。
- 使用 IP 直连时，如果未正确发送 SNI（Host 主机名），服务端会拒绝连接。

### 4) ReqResult / HTTP status / timeout

- ReqResult：网络是否连得通。
- HTTP status：服务端处理结果。
- timeout：长时间无响应的超时情况。

### 5) 排查策略

- 先分段定位：客户端->后端是否通，后端->外网是否通。
- 对比工具行为：Drogon 失败而 curl 成功，说明底层实现差异。
- 必要时抓包确认 DNS、TLS、SNI 与路由细节。

## 真实命令片段（节选）

```powershell
# 在服务端用 curl 验证外网连通性
curl.exe -I https://api.example.com
```

```powershell
# 强制 IPv4 验证（如果需要）
curl.exe -4 https://api.example.com
```

## 小结

- 直连失败不等于功能不可用，curl fallback 提供临时兜底。
- IPv6 优先 + SNI 问题是直连失败的常见原因。
- 排查要先分段，再对比工具行为。

## 自测问题

1. 为什么 Drogon 直连失败而 curl 能成功？
2. IPv6 优先可能导致什么问题？
3. SNI 在 TLS 握手里起什么作用？
4. ReqResult 与 HTTP status 的差异是什么？

## 答疑和自测问题答案

1. 可能是底层网络实现差异导致的直连失败（IPv6 路由/SNI/解析策略）。
2. 可能走到不可达的 IPv6 路径，导致连接失败。
3. SNI 告诉服务器你要访问哪个域名，否则共享 IP 的服务可能拒绝连接。
4. ReqResult 描述网络连通性；HTTP status 描述服务端处理结果。

## 下一步建议

- 如果要彻底修复直连：深入排查 DNS 解析策略与 SNI 发送逻辑。
- 若以交付为优先：保留 curl fallback，并记录技术债务。
