# 2026-02-01 概念学习笔记：二阶段核心架构与问题排查

本文档总结了今天关于 FinGuard Phase 2 开发过程中的核心问题、概念解释及生活化类比，旨在辅助理解项目逻辑与后续维护。

---

## 1. 核心问题：后端直连失败 (Drogon vs. Qwen)

### 【生活类比：餐厅订货与跑腿小弟】
*   **场景描述**：你开了一家餐厅（**后端服务**），客人在桌上点菜（**客户端**）。你平常都是通过店里的专用电话直接给肉菜供应商（**阿里云 Qwen API**）下订单。
*   **发生了什么**：现在店里的专用电话（**Drogon HttpClient 直连**）坏了，一打就报“号码错误/无法接通”（**BadServerAddress**）。
*   **解决方案**：为了不让客人饿肚子，你临时让店里的跑腿小弟（**curl 命令行程序**）带着现金跑去超市买菜回来。
*   **结果**：客人的菜还是端上桌了（**用户收到了 Qwen 的回复**），但因为要跑一趟，速度慢了一点，而且你还得付小弟跑腿费，店里的电话线也还没修好（**依赖降级路径，存在技术债务**）。

### 【技术要点】
*   **直连失败**：Drogon 库在底层建立 TCP/TLS 连接时在 DNS 解析或握手阶段受阻。
*   **降级机制 (Fallback)**：当主路径（C++ 库）失败时，程序调用系统的 `curl.exe` 作为备选方案完成请求。

---

## 2. 故障定位：客户端-后端 vs. 后端-外网

### 【生活类比：水管堵在哪？】
*   **核心逻辑**：
    *   **A段（客户端 → 后端）**：水从你家水龙头流到我家门口。
    *   **B段（后端 → 阿里云）**：我从我家去水厂拉水。
*   **现在的结论**：A段是通的（你能访问 `/health` 接口，SSE 也能断断续续出东西）。**堵在 B段**。
*   **为什么这么说**：因为我们在后端服务器上自己用 `curl` 都能接到水，但店里的水管（后端代码）接不到，说明是“后端到供水厂”的这段管子出问题了。

## 3. 个人工具的影响评估

### 【生活类比：私家车 vs. 运营公交】
*   **问题**：如果这个工具只是我自己用，这电话线（直连失败）修不修要紧吗？
*   **评估结果**：
    *   **私家车（个人自用）**：只要车能跑（功能闭环），空调不给力或者偶尔发动慢点（响应延迟）其实不影响你到目的地。
    *   **运营公交（商业上线）**：如果每天要拉几千个客人，靠跑腿小弟买菜（子进程开销）成本太高，且一旦小弟请假（系统资源紧张），整条线就瘫痪了。
*   **结论**：对于个人自用，**功能可用是最高准则**。目前 Phase 2 已经通过降级方案实现了功能闭环，达到了“能跑路”的状态。

## 4. 关键概念词典

### 1. SSE (Server-Sent Events)
*   **直白解释**：老师（服务器）在黑板上慢慢写字，你（客户端）盯着看，写一个字你读一个字，不用等老师整段写完。
*   **本项目应用**：Qwen 的回答是一个字一个字蹦出来的，让用户感觉反应很快。

### 2. DNS / SNI / TLS (排查的三座大山)
*   **DNS**：把网址翻译成 IP 地址的电话簿。
*   **SNI (Server Name Indication)**：你在敲别人家门时，得大声喊你要找的是谁（因为一栋楼里可能住着很多家，不喊名字网关不给开门）。如果直连报 `BadServerAddress`，很可能是名字没喊对。
*   **TLS**：加密传输的保安。如果你的保安（加密库）和对方的保安不合拍，连接也会被切断。

### 3. c-ares 与 IPv6 优先问题（今日新发现）
*   **c-ares**：一个异步 DNS 解析库，Drogon/trantor 用它来解析域名。
*   **问题**：c-ares 会优先返回 IPv6 地址，而某些 Windows 网络环境下 IPv6 路由不通。
*   **生活类比**：就像快递公司自动选择最近的分拣站，但那个站实际上在装修关门了（IPv6 路由不可达），导致你的包裹迟迟送不到。而如果你手动指定用老站（IPv4），就能正常送达。
*   **curl 为什么能工作**：curl 使用系统默认的 DNS 解析器，或者可以用 `-4` 强制使用 IPv4，避开了 IPv6 的坑。

### 4. Drogon HttpClient 的 SNI 限制
*   **问题**：当你用 IP 地址代替域名创建 HttpClient 时，Drogon 不知道原始的域名，无法在 TLS 握手时发送正确的 SNI。
*   **结果**：阿里云的服务器收到请求后不知道你要访问哪个服务（因为一个 IP 可能托管多个域名），直接拒绝连接。
*   **生活类比**：你去一个大厦敲门，门卫问"找谁"，你回答"我找门牌号 8.152.159.24"。门卫：？？？我只认名字不认号，滚。

### 5. Packet Capture (抓包/pktmon)
*   **直白解释**：在进出后端的路口装个监控摄像头，看看数据包到底有没有发出去，对方有没有回信息，有没有在半路被警察（防火墙）拦截。

## 5. 今天的学习心得
*   **功能闭环优先**：在直连遇到死代码（Deadlock）时，先用备选方案（Fallback）跑通功能，再回过头排查底层问题。
*   **多维度诊断**：不要只盯着代码看，要对比命令行工具（curl）、环境变量和网络抓包。
*   **分段排查**：分清是"客人进不来"还是"我出不去"，可以极大地减少无效工作。
*   **根因定位**：一旦发现"同一个功能，不同工具表现不同"，重点对比这些工具在底层实现上的差异（如 DNS 解析器、TLS 库、IPv4/IPv6 偏好等）。
*   **接受务实方案**：当修复成本过高（需要改上游库源码）而替代方案可行时，优先交付可用功能，将技术债务记录备忘。
