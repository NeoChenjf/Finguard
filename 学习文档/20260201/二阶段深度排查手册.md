# 二阶段技术难题深度解析：生活化通俗手册

为了帮助理解为什么我们的 AI 投资助手（FinGuard）在第二阶段遇到了“连接难题”，这篇文档用最直白的语言和生活中的例子来拆解那些高大上的技术名词。

---

## 1. 核心角色：Drogon 与 curl (内服药 vs 外用贴)

### 【Drogon HttpClient：店里的专用电话】
*   **它是啥**：它是我们程序内部自带的一个“拨号组件”。
*   **生活例子**：你开了一家店，店里装了一部**专线电话**，专门用来联系供应商订货。
*   **现状**：这部电话（Drogon）现在打给外面的阿里云（Qwen）时总是报“线路忙”或“号码错”，虽然电话本身没坏，但就是拨不通外线的某个特定号码。

### 【curl_fallback：临时找的跑腿小弟】
*   **它是啥**：它是一个独立的外部小程序（curl），专门负责在网络上爬取数据。
*   **生活例子**：店里电话坏了，为了不耽误生意，你雇了一个**跑腿小弟**（curl）。每次需要订货，你就写个纸条给小弟，让他跑去供应商那里买回来。
*   **现状**：小弟跑得很稳，能买到货（Qwen 的回复），但每次都要重新起跑，比打电话稍微慢一点，且不如电话（内嵌代码）管理起来方便。

---

## 2. 罪魁祸首：IPv4 与 IPv6 (老路 vs 未完工的高速)

### 【IPv4：虽然拥挤但修好的老省道】
*   **特点**：老牌、稳定、哪都能去。
*   **现状**：绝大多数互联网服务（包括阿里云）都完美支持。

### 【IPv6：宽敞但出口没修好的新高速】
*   **特点**：车道极多（地址多），是未来的趋势。
*   **现状**：虽然地图上标好了这部分路，但**你家门口到这条高速的引桥还没修好**（由于网络服务商或系统配置原因，本机的 IPv6 路由不可达）。

### 【发生什么了？】
当我们的程序问：“去阿里云怎么走？”
*   **GPS（DNS 解析）** 告诉程序：“走 IPv6 新高速吧，快！”
*   **程序** 听话地去了，结果发现高速入口封死着（路由不可达），直接导致“连接失败”。
*   **而 curl 小弟** 比较老练，他默认走老路（IPv4），所以他能到。

---

## 3. 幕后黑手：c-ares (执着的导游)

### 【c-ares：只看路牌不看路况的导游】
*   **角色**：它是程序里的 DNS 解析库，负责把“网址”翻译成“地图坐标”。
*   **它的问题**：它是一个极其推崇 IPv6 的导游。每次它都优先把 IPv6 的坐标塞给程序，哪怕那条路根本不通。
*   **结果**：即使我们想走 IPv4，这个导游也会固执地把我们带向 IPv6 的断头路。

---

## 4. 修复失败的原因：SNI (进大楼必须喊名字)

### 【SNI：共享办公室的门卫】
*   **生活例子**：阿里云的服务器像一栋巨大的**共享写字楼**，里面住着成千上万个公司（不同的 API 服务）。
*   **进门规则**：你敲开大门口（建立连接）时，保安会问：“你找谁？”你必须清晰喊出“我找阿里云公司”（即 Host 主机名），保安才让你进。
*   **我们的困境**：
    *   为了绕过那个“迷路导游”（c-ares），我们尝试直接拿坐标（IPv4 地址）去找阿里云。
    *   结果到了门口，保安问“找谁”，因为我们的程序（Drogon HttpClient）暂时没学会“对着地址喊名字”，保安（阿里云网关）就直接把我们轰出来了（拒绝连接）。

---

## 5. 总结：目前的 Phase 2 状态

这就好比：
1.  **目标**：要把货（AI 建议）从供货商（阿里云）运到店里。
2.  **主线方案**：用店里电话（Drogon）订货，但因为导游（c-ares）非要把线路引向还没通的 IPv6，且直接拿坐标去又会被保安（SNI）拦下，暂时**卡住了**。
3.  **应急方案**：雇个跑腿小弟（curl_fallback）。虽然稍微费点事，但小弟认路，能把货安全拿回来。

**目前结论**：我们先让“跑腿小弟”干活，确保店能开门营业（Phase 2 功能闭环）。修复电话线（Drogon 直连）作为一个长期维修任务，不影响我们现在开始装修柜台（Phase 3 规则引擎开发）。
