# study.md - config 中的 YAML 是什么

## 标题与目标
用最通俗的方式理解 YAML 在配置文件中的含义，并能读懂项目里的 `rules.yaml`。

## 学习路线 / 分层
1. 入门：YAML 是什么、用来干什么。
2. 进阶：缩进、键值、数组的基本规则。
3. 实战：看懂本项目 `finguard/config/rules.yaml` 的结构。

## 核心概念解释
### 通俗版（生活类比）
把 YAML 想成“家庭收纳清单”或“菜单配方单”。
- 每一行像一张便签：**名称：内容**。
- 缩进表示“属于谁的”：就像把小物件放在同一个抽屉里。
- 列表就是“清单”，一条一条列出来。

### 专业版（准确解释）
YAML 是一种**人类可读的数据序列化格式**，常用于配置文件。
- **键值对**表示字段与值（例如 `gold_percent: 0.10`）。
- **缩进**表示层级结构（父子关系）。
- **列表**用 `-` 表示多个条目。
- YAML 的目标是让配置易读、易改、易维护。

## 关键流程 / 步骤
1. 打开配置文件（例如 `finguard/config/rules.yaml`）。
2. 识别顶层键（如 `allocation`、`risk_limits`、`warnings`）。
3. 根据缩进判断层级与归属关系。
4. 需要调整规则时，只改对应字段，不改缩进结构。

## 真实代码片段
下面是本项目配置的简化示例：
```yaml
allocation:
  gold_percent: 0.10
  bonds:
    method: age_decade
    default_age: 35
  equities:
    index_allocation:
      etfs:
        us: "VOO"
```
解释：
- `allocation` 是“资产配置”的总抽屉。
- `gold_percent` 是黄金比例。
- `bonds` 里的 `method` 表示债券的计算方式。
- `equities` 里 `index_allocation` 指股票指数配置。

## 小结 + 自测题
小结：YAML 就是“结构化清单”，通过缩进表达层级，通过键值表示规则。
自测题：
1. YAML 为什么强调缩进？
2. `-` 在 YAML 中表示什么？
3. 如果要改黄金比例，应该改哪个字段？

## 下一步建议
- 阅读完整的 `rules.yaml`，标出你最想调整的 3 个字段。
- 尝试在副本中修改一个比例，并观察结构是否仍然清晰。

# 追加主题：问卷提交位置与聊天接口引用方式

## 标题与目标
理解“问卷随请求提交”与“问卷保存到用户档案再引用”的区别，并能选择合适方案。

## 学习路线 / 分层
1. 入门：认识两种提交方式的差异。
2. 进阶：理解它们对性能、复用、维护的影响。
3. 实战：看懂请求体里的字段写法。

## 核心概念解释
### 通俗版（生活类比）
- **随请求提交**：每次点外卖都重新填地址和口味。
- **保存到档案**：先在会员中心保存地址，点外卖时只选“家庭地址”。

### 专业版（准确解释）
- **随请求提交（内联）**：问卷字段直接放在 `/api/v1/chat/stream` 请求体内，规则引擎即时使用。
- **档案引用（持久化）**：先把问卷写入用户档案（文件/数据库），聊天请求只传 `profile_id` 或通过账号绑定读取档案。

## 关键流程 / 步骤
### 方式 A：随请求提交
1. 客户端在聊天请求体里带 `questionnaire`。
2. 服务端直接读取并判定规则。
3. 每次请求重复传输同样字段。

### 方式 B：档案保存 + 聊天引用
1. 用户提交问卷到档案接口（或更新档案）。
2. 服务端把问卷存到本地文件或数据库。
3. 聊天请求只带 `profile_id` 或账号。
4. 服务端读取档案，再做规则判定。

## 真实代码片段
方式 A（内联）：
```json
{
  "prompt": "......",
  "questionnaire": {
    "age": 35,
    "investor_profile": "experienced",
    "individual_stock_percent": 0.30
  }
}
```

方式 B（档案引用）：
```json
{
  "prompt": "......",
  "profile_id": "NeoChen"
}
```

## 小结 + 自测题
小结：
- 内联方式简单，但每次重复。
- 档案方式更适合长期使用与多端同步。

自测题：
1. 为什么档案方式更适合长期使用？
2. 内联方式最大的优点是什么？
3. 什么时候必须用档案方式？

## 下一步建议
- 若是个人使用：先用本地文件存档，后续再接数据库。
- 若是多端使用：优先使用档案方式，并绑定账号作为 `profile_id`。

## 项目落地约定（本项目）
- 使用 Header `X-User-Id` 作为账号标识。
- 问卷通过独立接口写入档案（`/api/v1/profile/upsert`）。
- 当用户说“教教我”时，默认更新当前教学文档（本文件），除非用户明确要求新建。

# 追加主题：YAML 缩进与列表、档案与 LLM 上下文

## 标题与目标
厘清 YAML 的缩进与列表规则；理解“档案在后端”与“LLM 上下文”的关系。

## 学习路线 / 分层
1. 入门：YAML 是给人读的，但机器怎么用。
2. 进阶：缩进如何定义层级；`-` 如何形成列表。
3. 实战：档案保存与 LLM 是否“知道”的区别。

## 核心概念解释
### 通俗版（生活类比）
- YAML 像“收纳清单”：**人看懂**就能改；**机器看懂**是因为有“翻译员”（解析器）。
- 缩进像“抽屉层级”：放在同一层的东西属于同一个抽屉。
- `-` 像“清单的每一项”：同一缩进的一组 `-` 就是一份清单。

### 专业版（准确解释）
- YAML 是**人类可读的序列化格式**，机器通过**YAML 解析器**把它变成“键值映射（map）/列表（sequence）/标量（scalar）”的数据结构。
- **缩进决定父子关系**：缩进越深，层级越低（更具体）。
- **`-` 表示列表项**：同一缩进下的 `-` 构成一个列表，列表整体归属于它的父键。

## 关键流程 / 步骤
1. 机器读取 YAML 文件内容（字节流）。
2. YAML 解析器把文本解析成“树状结构”（map/sequence/scalar）。
3. 程序根据键路径读取值（如 `allocation.gold_percent`）。
4. 规则引擎用这些值进行校验或生成建议。

## 真实代码片段
下面示例展示缩进层级与列表归属：
```yaml
allocation:
  gold_percent: 0.10
  bonds:
    method: age_decade
  equities:
    index_allocation:
      etfs:
        us: "VOO"
        china_a: "沪深300"
      ratios:
        china_to_us: [1, 4]
      notes:
        - "按全球资本市场比例"
        - "中国内部再细分"
```
解释：
- `allocation` 是顶层键。
- `gold_percent` 属于 `allocation`。
- `notes` 下的两个 `-` 是**同一个列表**的两个条目。

## 小结 + 自测题
小结：
- YAML 既给人看，也给机器用；机器通过解析器把文本变成数据结构。
- 缩进决定层级；`-` 表示列表项。
- 列表项属于它们共同的父键（同一缩进）。

自测题：
1. YAML 为什么强调缩进？（提示：层级）
2. `-` 在 YAML 中表示什么？（提示：列表项）
3. 为什么“档案在后端”不等于“LLM 自动知道”？（提示：上下文）

## 关键误区纠正（结合你的理解）
- 你说“缩进判断层级”是对的：**缩进就是层级**。
- 你说“- 代表多个条目，且它们归属同一小标题”也是对的：  
  **同一缩进下的 `-` 属于同一个列表，这个列表属于同一个父键**。  
  但也可以有**顶层列表**（`-` 顶格时，表示文件本身就是列表）。
- “黄金比例在 allocation 下的 gold_percent 修改”是正确的。

## 档案 vs LLM 上下文（你的疑问）
- 档案只在**你的后端**有效；LLM 本身不会“自动知道”档案。  
- 如果你希望 LLM 了解用户档案：  
  1) 在请求 prompt/system 中**显式加入档案摘要**；或  
  2) 在服务端先用规则引擎生成结论，再把结论加入 prompt。  
- 每次请求默认是**无状态**的，除非你把上次的上下文/档案注入进去。

## 下一步建议
- 如果只做规则校验：档案保存在后端即可，不需要传给 LLM。
- 如果需要个性化回答：建议用“精简档案摘要”注入到 system prompt。
