# 优化实施入门（配置缓存与 HttpClient 复用取舍）

## 标题与目标

目标：把你列出的每个改动点都讲清楚“改动逻辑”。你看完后能回答：改了哪里、为什么改、收益是什么、为什么第二个优化最后回退。

## 学习路线/分层

1. 先理解优化 1：配置缓存（5 秒 TTL）是怎么把热路径 I/O 拿掉的。
2. 再逐文件看改动逻辑：`reliability_config.*`、`routes.cpp`、`llm_client.cpp`。
3. 再理解优化 2：为什么“HttpClient 复用”理论正确但在当前同步模型里反而退化。
4. 最后看工程决策：为什么暂时保留 per-request client。

## 核心概念解释（带生活类比）

- 热路径：每个请求都会走的路径，哪怕只多 1ms，在高并发下都会被放大。
- TTL（Time To Live）：缓存有效期。这里是 5 秒，表示 5 秒内复用旧配置。
- 缓存命中：不读磁盘，直接用内存中的配置。
- 缓存失效：超过 TTL，再读一次文件刷新缓存。

生活类比：
- 原来每个客人来都要去仓库翻一次“运营手册”（文件 I/O）。
- 现在改成前台桌上放一份“5 秒更新一次”的手册（内存缓存）。

## 关键流程/步骤（可复刻）

1. 先识别热点
   - 每请求反复 `load_*_config()`，磁盘 I/O + YAML/JSON 解析过多。

2. 实施优化 1（配置缓存）
   - 在 util 层实现统一缓存器。
   - `util` = utility（通用工具层），放跨模块可复用能力（如配置读取、限流、超时、断路器配置）；本次缓存放在 `src/util/reliability_config.*`，避免在 routes/llm 重复实现。
   - 在 routes 和 llm 热路径把 `load_*` 改成 `cached_*`。
   - 把 LLM 配置也做 5 秒缓存，避免每次 `load_config()`。

3. 实施优化 2（HttpClient 复用试验）
   - 试 `static` 共享、再试 `thread_local`。
   - 用基线指标验证吞吐和延迟。

4. 基于数据做取舍
   - 发现同步 `sendRequest()` + 共享 client 出现序列化效应，吞吐大幅下滑。
   - 回退该项，保留 per-request client，并补注释说明原因。

## 每个改动点的逻辑（逐文件拆解）

### 1) `src/util/reliability_config.h`

改动：新增 4 个函数声明：
- `cached_rate_limit_config()`
- `cached_timeout_config()`
- `cached_circuit_breaker_config()`
- `cached_concurrency_config()`

逻辑：
- 给业务层一个“安全替代入口”。
- 业务代码不需要知道缓存实现细节，只需要把 `load_*` 替换成 `cached_*`。

### 2) `src/util/reliability_config.cpp`

改动：新增 `ConfigCache<T>` 模板、4 个全局缓存实例、TTL=5 秒、`std::mutex` 保护。

逻辑：
- `ConfigCache<T>::get(loader)` 做两件事：
  1. 没过期：直接返回内存值。
  2. 过期：加锁后调用 `loader()` 重新读文件并更新。
- `std::mutex` 保证并发线程不会同时刷新同一份配置，避免竞态与重复 I/O。
- 5 秒 TTL 是平衡点：
  - 太短：I/O 仍然频繁。
  - 太长：配置变更生效太慢。

### 3) `src/server/routes.cpp`

改动：路由里的 `load_rate_limit_config()` / `load_concurrency_config()` 等热路径调用换成 `cached_*`。

逻辑：
- `/plan`、`/chat/stream` 是高频路径。
- 这里每少一次文件读取，系统就少一次磁盘和解析开销。
- 你提到的 `client.load_config()` 相关检查也做了 5 秒缓存，避免每个请求重复读 `llm.json`。

### 4) `src/llm/llm_client.cpp`

改动 A：`stream_chat()` 中配置读取改为 `cached_rate_limit_config()` / `cached_timeout_config()` / `cached_circuit_breaker_config()`。

逻辑：
- 把原本每请求重复读取的策略配置改为“周期刷新”。
- 直接降低文件 I/O 次数和解析对象创建次数。

改动 B：`LlmConfig` 使用静态缓存（5 秒 TTL）。

逻辑：
- `api_key`、`api_base` 等配置不需要每请求读盘。
- 用 `static + mutex + loaded_at` 在函数内部做轻量缓存，减少 `load_config()` 触发频率。

## 优化 2：HttpClient 复用为什么“验证后回退”

你关心的点：既然复用通常是好事，为什么这里回退？

核心逻辑：
- 当前调用是同步 `sendRequest()`。
- 在共享 HttpClient 上，请求容易出现串行化行为（并发度被压扁）。
- 所以虽然“少了建连开销”，但“并发能力下降”带来的损失更大。

`thread_local` 为什么也不理想：
- 它把共享粒度从“全局 1 个”改成“每线程 1 个”。
- 但可用并发上限仍受 IO 线程数约束，不足以覆盖 chat 场景吞吐目标。

工程决策：
- 当前阶段优先吞吐与延迟，保留 per-request client。
- 在代码里加注释，明确这是“有意为之”，不是忘记优化。
- 下一步若要继续优化，要改架构到异步请求模型，而不是只改对象复用策略。

## 真实代码片段（带逐行注释）

### 1) 配置缓存核心模板（简化版）

```cpp
template <typename T>
struct ConfigCache {
    std::mutex mtx;                  // 保护并发访问
    T value{};                       // 缓存的配置值
    Clock::time_point loaded_at{};   // 上次刷新时间

    template <typename Loader>
    T get(Loader loader) {
        std::lock_guard<std::mutex> lock(mtx);            // 防止并发刷新
        const auto now = Clock::now();                    // 读取当前时间
        if (now - loaded_at > kCacheTtl) {                // 超过 TTL 则刷新
            value = loader();                             // 重新读文件并解析
            loaded_at = now;                              // 更新时间戳
        }
        return value;                                     // 返回缓存值
    }
};
```

### 2) 路由热路径替换示意

```cpp
// 旧：每请求都读配置
// const auto rate_cfg = util::load_rate_limit_config();

// 新：优先走缓存
const auto rate_cfg = util::cached_rate_limit_config();   // 减少磁盘 I/O
const auto conc_cfg = util::cached_concurrency_config();  // 减少重复解析
```

### 3) LLM 请求路径中的取舍注释（思路）

```cpp
// 共享 HttpClient + 同步 sendRequest 在当前链路会降低并发
// 所以当前保留 per-request client，优先保障 chat 场景吞吐
auto client = drogon::HttpClient::newHttpClient(host_url); // 每请求创建
```

## 小结 + 自测问题（含答案）

小结：
- 优化 1 的本质是把“每请求读盘”改成“按 TTL 刷新”。
- 这不是改业务规则，而是改读取策略，收益是直接降低热路径 I/O。
- 优化 2 在当前同步架构里出现反效果，说明“理论优化”必须以压测数据为准。
- 回退不是失败，而是正确的工程决策：先保吞吐，再做架构级改造。

自测题：
1. 为什么要在 `.h` 里新增 `cached_*` 声明，而不是只改 `.cpp`？
2. `ConfigCache<T>` 里的 `mutex` 解决了什么问题？
3. 为什么共享 HttpClient 可能导致 chat 吞吐下降？
4. “验证后回退”体现了什么工程原则？

答案：
1. 因为业务调用方需要稳定接口，才能把 `load_*` 统一替换成 `cached_*`。
2. 防止并发线程同时刷新缓存，避免竞态和重复 I/O。
3. 在同步请求模型里可能发生请求串行化，并发能力下降。
4. 数据驱动决策：以指标结果而不是“常识偏好”决定是否保留优化。
