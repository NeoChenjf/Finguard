# 基线压测执行入门（YAML 解析失败与 BOM 修复）

## 标题与目标

目标：理解这次基线压测里“为什么会全部 429”，并掌握完整链路：发现问题 -> 定位根因 -> 修复配置编码 -> 复测验证。

## 学习路线/分层

1. 先看现象：为什么压测一上来全是 429。
2. 再看因果：429 与限流配置的关系。
3. 再看根因：为什么 YAML 解析失败会回退默认值。
4. 最后看修复：为什么用无 BOM 写入 `build/config/` 能解决。

## 核心概念解释（带生活类比）

- 基线压测（Baseline）：先测系统“原始状态”的性能，不先做优化。
- 429：服务在说“请求太多了，请稍后再来”。
- 配置回退默认值：读取你写的配置失败后，程序用内置默认值继续跑。
- UTF-8 BOM：文件开头的 3 个字节标记（`EF BB BF`）。部分解析器会把它当成异常输入。
- YAML 解析失败：配置文件内容或编码格式不符合解析器预期，导致读取失败。

生活类比：
- 你给厨房写了新排班表（高配额），但排班表第一页有隐藏脏字符（BOM）。厨师读不懂就按旧排班（默认低配额）干活，结果客人一多就全部被拦（429）。

## 关键流程/步骤（可复刻）

1. 发现异常
   - 基线压测几乎全部 429，和“压测宽松值”预期不一致。

2. 先验证是不是限流在工作
   - 429 通常首先怀疑限流参数太小。

3. 追查配置是否真的生效
   - 你期望的是 `rate_rps=5000`，但实际表现像 `rate_rps=5`。
   - 这说明很可能配置没读成功，程序回退默认值。

4. 检查配置编码
   - 发现 YAML 文件是 UTF-8 BOM 编码。
   - 当前解析链路对 BOM 不兼容，触发 YAML 解析失败。

5. 修复写入方式
   - 用 `[System.IO.File]::WriteAllText()` + `UTF8Encoding($false)` 写入无 BOM 文件。
   - 写入位置是 `build/config/`，因为运行时读取的是这个目录下的配置。

6. 重启并复测
   - 重启服务后再跑 S1-S4，确认 429 消失，指标恢复正常。

## 真实命令/代码示例（带逐行注释）

### 1) 快速检查是否存在 BOM（PowerShell）

```powershell
# 读取配置文件原始字节
$path = "d:\AI_Investment\finguard\build\config\rate_limit.yaml"
$bytes = [System.IO.File]::ReadAllBytes($path)
# 打印前 3 个字节（如果是 EF BB BF，说明有 UTF-8 BOM）
"{0:X2} {1:X2} {2:X2}" -f $bytes[0], $bytes[1], $bytes[2]
```

说明：
- 输出 `EF BB BF` 代表文件带 BOM。
- 这个隐藏头在部分 YAML 解析链路中会导致解析失败。

### 2) 用无 BOM 方式写配置到 `build/config/`

```powershell
# 要写入的配置文本（示例）
$yaml = @"
rate_rps: 5000
capacity: 10000
max_inflight: 1000
"@

# 目标路径：运行时读取目录
$path = "d:\AI_Investment\finguard\build\config\rate_limit.yaml"

# 创建“不带 BOM 的 UTF-8”编码器
$utf8NoBom = New-Object System.Text.UTF8Encoding($false)

# 写入文件（覆盖）
[System.IO.File]::WriteAllText($path, $yaml, $utf8NoBom)
```

说明：
- `UTF8Encoding($false)` 里的 `false` 表示“不写 BOM”。
- 这样能避免解析器把开头 BOM 当成脏输入。

### 3) 修复后复测（示意）

```powershell
# 重新运行基线压测（示例）
python d:\AI_Investment\tools\p5_bench.py --scenario s1
python d:\AI_Investment\tools\p5_bench.py --scenario s2
```

说明：
- 重点不是命令名，而是“修复后 429 是否明显下降到 0 或接近 0”。
- Phase5 记录里，修复后 S1-S4 错误率回到 0%。

## 小结 + 自测问题（含答案）

小结：
- 这次问题不是压测工具错误，而是配置文件编码触发了 YAML 解析失败。
- 解析失败后程序回退默认限流值（`rate_rps=5`），导致压测全 429。
- 关键修复点是：把配置以“UTF-8 无 BOM”写到运行时实际读取目录 `build/config/`。
- 修复必须靠复测闭环验证，不能只看“文件写成功”。

自测题：
1. 为什么“全部 429”会优先怀疑限流配置？
2. BOM 是什么，它为什么可能引发解析问题？
3. 为什么要写到 `build/config/` 而不是只改源码目录？
4. `UTF8Encoding($false)` 里的 `false` 代表什么？

答案：
1. 429 的语义就是“请求过多”，最直接对应限流参数。
2. BOM 是文件头字节标记；解析链路不兼容时会把它当异常输入。
3. 因为进程运行时读取的是 `build/config/`，改错目录不会生效。
4. 表示写 UTF-8 时不附加 BOM。

## 下一步建议

1. 给配置加载阶段加启动日志，明确打印“配置来源路径 + 是否加载成功”。
2. 在压测脚本前加一次配置自检（路径、编码、关键参数）。
3. 把“无 BOM 写入”封装成统一脚本，避免重复踩坑。
