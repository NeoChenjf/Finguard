# 2026-02-06 学习总览：RuleEngine 请求/响应侧流程

## 标题与目标

读完这份文档，你能定位 RuleEngine 的代码位置，说明请求侧与响应侧各做什么检查，并完整描述 `/api/v1/chat/stream` 的执行流程。

## 整体学习路线

1. 入口路由与数据来源：prompt、问卷档案、规则配置。
2. RuleEngine 配置：默认值与 `rules.yaml` 的实际生效字段。
3. 请求侧检查：关键词与画像限制。
4. 响应侧检查：结构、内容、数值、合规。
5. SSE 输出：token / cite / metric / warning / done。

## 核心概念解释（带类比）

把 RuleEngine 当作“出门前的双重检查清单”。
- **请求侧**像“进门安检”：先看你带了什么（prompt 关键词、问卷画像），该提示就提示。
- **响应侧**像“成品质检”：报告是否按模板写、数值是否合规、是否出现禁用措辞。

解释：

- **请求侧**：检查“你发给服务端的输入”，主要是 prompt 文本 + 问卷画像（questionnaire）。在调用 LLM 之前触发。
- **响应侧**：检查“LLM 返回的完整报告文本”，在 LLM 输出完成后触发，用于结构/内容/数值/合规校验。

## 关键流程与步骤

### 1) 代码入口与文件位置

- RuleEngine 定义与实现：`finguard/src/risk/rule_engine.h`、`finguard/src/risk/rule_engine.cpp`。
- 规则配置：`finguard/config/rules.yaml`（仅部分字段会被解析）。
- 入口路由与集成：`finguard/src/server/routes.cpp` 中的 `/api/v1/chat/stream`。
- 问卷档案读写：`finguard/src/risk/profile_store.*`。

### 2) 请求侧逻辑（check_request）

触发点：`routes.cpp` 中拿到 `prompt` 与 `questionnaire` 后调用 `RuleEngine::check_request()`。

输入形态说明：
- `prompt` 是纯文本字符串。
- `questionnaire` 是 JSON（`Json::Value`）。
- `rules.yaml` 只是配置来源，不是被检查对象本身。

主要检查：
1. **画像字段是否缺失**：`investor_profile`、`age`、`individual_stock_percent` 若缺失会产生警告，并使用默认值。
2. **关键词命中**：`rules.yaml` 中 `warnings.keyword_rules.forbidden_keywords`，命中后生成风险提示文案。
3. **画像限制**：如 novice 不允许个股配置、或个股比例超过上限会产生告警。
4. **最小占比阈值**：个股占比低于 `min_single_asset_percent` 触发警告。

> 注意：如果问卷不存在，仍会继续执行，但会追加 `profile_missing_or_unreadable` 等告警。

### 3) 响应侧逻辑（check_response）

触发点：LLM 输出完成后，`routes.cpp` 调用 `RuleEngine::check_response(result.full_text)`。

输入形态说明：
- `response_text` 是 LLM 输出的完整纯文本字符串。
- 检查的是“文本内容”，不是 JSON 或 YAML。

主要检查：
1. **结构检查**：必须包含固定章节（如“公司信息”“财务状况…”）。
2. **内容检查**：公司信息字段、财务指标等是否缺失。
3. **数值阈值**：PEG、负债率、ROE 是否超过阈值（PEG<=2、负债率<=50%、ROE>20%）。
4. **语义依据**：出现“偏高/偏低”等判断时必须有数值或阈值依据。
5. **结论要点数量**：结论区 1-3 条要点。
6. **合规声明**：必须包含来源信息。
7. **禁用措辞**：如“买入/卖出/加仓”等，出现即报 error 级别告警。

### 4) 规则配置生效范围

RuleEngine 并不会解析 `rules.yaml` 的所有字段，当前实际生效的字段包括：
- `allocation.gold_percent`
- `allocation.bonds.default_age`
- `risk_limits.min_single_asset_percent`
- `stock_selection.peg_max`
- `stock_selection.debt_ratio_max`
- `stock_selection.roe_min`
- `warnings.message_template`
- `warnings.keyword_rules.forbidden_keywords`
- `allocation.equities.individual_stock.profile_limits.*`

其他字段属于配置预留或供后续扩展使用。

### 5) SSE 输出流程（整体链路）

1. 读取 `prompt`。
2. 读取问卷档案（`X-User-Id`）。
3. 加载规则配置并执行 **请求侧**检查。
4. 调用 LLM 获取流式结果。
5. 执行 **响应侧**检查。
6. 组装 SSE：`token` → `cite` → `metric` → `warning` → `done`。

自然语言解释（这几类事件“在说什么”）：
- **token**：模型正在“逐字输出正文内容”。客户端拿到 token 就能边接收边显示。
- **cite**：引用来源占位或真实来源标识（没有来源时会是 `none`）。
- **metric**：这次生成的用量统计（prompt/完成/总 token 数）。
- **warning**：规则引擎给出的检查结果（结构缺失、数值超标、合规禁词等）。
- **done**：本次 SSE 流结束的明确标记。

### 6) SSE 输出示例（片段）

> 说明：每条 SSE 事件是一行 `data: <json>`，后跟一个空行。

```text
data: {"type":"token","payload":"【公司信息】\n- 公司名称：TechCorp\n- 交易所或代码：TSC\n- 行业或赛道：信息技术\n- 核心产品或服务：云服务\n\n"}

data: {"type":"cite","payload":"none"}

data: {"type":"metric","payload":{"prompt_tokens":123,"completion_tokens":456,"total_tokens":579}}

data: {"type":"warning","payload":{"code":"CONTENT.MISSING_COMPANY_FIELD","section":"公司信息","reason":"内容缺失（003）公司信息缺少字段：公司名称","basis":"阈值=必须包含公司信息字段；来源=模板要求","evidence":"未提供","trigger_terms":[],"severity":"warn"}}

data: {"type":"warning","payload":{"code":"NUMERIC.THRESHOLD_BREACH","section":"财务状况（定量指标，含好价格判断）","reason":"数值超标（001）指标超出阈值：PEG","basis":"阈值=PEG<=2；来源=守拙价值多元化基金理念","evidence":"2.7","trigger_terms":["PEG 2.7"],"severity":"warn"}}

data: {"type":"done","payload":"done"}
```

### 7) 示例的“人话解读”

把上面的片段读成一句话就是：
“系统先把报告正文一点点吐出来（token），然后补一句‘引用来源目前没有’（cite=none），再告诉你这次生成大概用了多少 token（metric）。接着它说：检查结果里发现了两个问题（warning），一个是公司信息缺字段，另一个是 PEG 超过阈值。最后发一个 done 告诉你这次流结束了。”

## 小结

- RuleEngine 的请求侧在进入 LLM 前执行，用于提示高风险内容与画像限制。
- 响应侧在 LLM 输出后执行，用于保证结构、内容与合规性。
- `/api/v1/chat/stream` 是整条链路的汇合点，负责串联检查与 SSE 输出。

## 自测问题

1. RuleEngine 的请求侧和响应侧分别在什么时机触发？
2. 哪些 `rules.yaml` 字段是当前代码真正读取的？
3. 响应侧有哪些“结构类”的强制检查？
4. 为什么禁用措辞会被标记为 error？

## 自测问题答案

1. 请求侧在调用 LLM 前执行；响应侧在 LLM 完整输出后执行。
2. 见“规则配置生效范围”小节中列出的字段。
3. 必须包含固定章节；结论要点数量 1-3；合规声明包含来源。
4. 因为涉及合规风险，需要最高级别提示。
