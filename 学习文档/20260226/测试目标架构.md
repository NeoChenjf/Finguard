# 测试目标架构与静态库方案讲解（Phase 6）

## 标题与目标

目标：搞清楚“静态库核心源文件”真正含义，并理解为什么 Phase 6 推荐 `finguard_core` 静态库方案。

## 学习路线/分层

1. 先纠正一个常见误解：静态库 ≠ 代码不会变。  
2. 再看静态库到底“静态”在什么地方。  
3. 最后看你项目为什么要这么拆。  

## 核心概念解释（带类比）

- 你问的这句理解需要修正：  
  “静态库核心源文件”**不是**“这部分以后基本不改”。  
  正确理解：这部分是“可被多个目标复用的核心实现”，通过静态链接打包复用。

- `静态库`里的“静态”，主要指**链接方式**：  
  编译后先生成 `.lib/.a`，再把库代码拷贝进可执行文件。  
  它不是“逻辑静态不变”的意思，代码照样可以改、照样会迭代。

- 为什么叫“核心源文件”：  
  因为这些文件承载业务逻辑（规则引擎、限流、熔断、配置加载、路由等），主程序和测试都依赖它们。  
  类比：把发动机做成标准模块，轿车（主程序）和测试台（gtest）都用同一台发动机。

## 关键流程/步骤（可复刻）

1. 把 `main.cpp` 以外的核心实现放入 `finguard_core`。  
2. 主程序 `finguard` 只负责入口，链接 `finguard_core`。  
3. 测试程序 `finguard_tests` 也链接 `finguard_core`。  
4. 这样测试测到的就是“真实生产代码”，不再是占位测试。  

## 真实代码片段（逐行注释）

```cmake
add_library(finguard_core STATIC                 # 定义静态库目标：finguard_core
    src/llm/llm_client.cpp                       # 核心逻辑文件（示例1）
    src/risk/rule_engine.cpp                     # 核心逻辑文件（示例2）
    src/server/routes.cpp                        # 核心逻辑文件（示例3）
)

add_executable(finguard src/main.cpp)           # 主程序入口
target_link_libraries(finguard PRIVATE          # 主程序链接核心静态库
    finguard_core
)

add_executable(finguard_tests                   # 测试可执行文件
    tests/test_rules.cpp
    tests/test_cache.cpp
)
target_link_libraries(finguard_tests PRIVATE    # 测试链接同一份核心代码
    finguard_core
    GTest::gtest_main
)
```

## 静态库方案的优点与边界

- 优点：
  - 避免主程序和测试各自复制/编译一套逻辑，减少重复与不一致。  
  - 测试直接覆盖真实核心实现，回归发现更早。  
  - CMake 结构更清晰：入口层（exe）与业务层（core）分离。  

- 边界：
  - 静态库不代表“不会改”；改了核心代码仍需重新编译和测试。  
  - 静态库也不自动保证“安全”，安全来自：测试、代码评审、边界校验、运行监控。  

## 小结 + 自测题（含答案）

小结：
- “静态”是链接方式，不是“逻辑不变”。  
- `finguard_core` 的价值是复用与可测性，不是冻结改动。  
- Phase 6 的架构调整核心目的是让 gtest 真正测到生产核心代码。  

自测题：
1. 静态库里的“静态”到底指什么？  
2. 为什么 `finguard_tests` 要链接 `finguard_core`？  
3. 仅有静态库是否就等于代码安全？  

答案：
1. 指链接方式（编译产物在链接时打包进可执行文件）。  
2. 为了让测试直接验证真实核心逻辑，防止“测了但没测到关键代码”。  
3. 不等于；还需要测试、评审、校验和监控共同守护。
