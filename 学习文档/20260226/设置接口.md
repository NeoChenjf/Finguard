# Phase 6 设置接口概念讲解

## 标题与目标

目标：用最短路径理解你提到的 7 个点：`Release 构建`、`gtest`、`GET/POST /api/v1/settings`、`temperature`、`timeout_ms`、`mutex 写锁`。

## 学习路线/分层

1. 先看“为什么要做”（Phase 6 的目标）。
2. 再看“接口做什么”（GET/POST settings）。
3. 再看“参数是什么意思”（temperature、timeout_ms）。
4. 最后看“并发安全怎么保证”（mutex 加锁写文件）。

## 核心概念解释（带生活类比）

- `Release 构建`：给线上/交付用的“性能版”编译，开启优化（如 `/O2`）。  
  其中 `/O2` 是 MSVC 的“高性能优化”编译选项，会让编译器更激进地做内联、循环与指令层优化，目标是让程序跑得更快（但调试体验通常不如 Debug 直观）。  
  类比：Debug 像“带教学字幕的练习车”，Release 像“正式上路的量产车”。

- `gtest`：GoogleTest，C++ 常用单元测试框架，用来自动验证代码行为。  
  类比：每次改代码前后都跑一套“体检项目”，防止回归。

- `GET /api/v1/settings`：读取当前可展示配置（你的理解是对的），用于前端页面初始化显示。

- `POST /api/v1/settings`：提交并更新配置（你的理解也是对的），用于用户在设置页点击保存时写入新值。

- `temperature`（number）：控制模型输出“随机性/发散度”。  
  - 低值（如 0.1~0.3）：更稳定、更保守、重复性更高。  
  - 高值（如 0.8~1.2）：更发散、更有创造性，但波动也更大。  
  类比：做菜放调料的“自由度旋钮”。

- 这里说的“模型”是指你后端通过 `api_base` 调用的**大语言模型服务**，例如：
  - 阿里 DashScope 兼容接口下的 `qwen-plus`
  - OpenAI 兼容接口下的 `gpt-4o-mini`
  - 其他兼容 OpenAI API 协议的模型
  也就是说，这个“模型”不是 C++ 编译模型，而是“你发起 HTTP 请求的下游 AI 模型”。

- `temperature` 在**运行时请求模型时**使用，不是在编译时使用：
  - 不是编译阶段：`cmake` / `Release` / `/O2` 和 `temperature` 没关系。
  - 是配置阶段：你可在 `config/llm.json` 或 `POST /api/v1/settings` 里设置它。
  - 是调用阶段：当 `/plan`、`/chat/stream` 等接口真正向下游模型发请求时，后端把 `temperature` 放进请求 JSON，一起发给模型。
  - 是生效阶段：模型按这个值生成回复，所以它影响“回答风格”，不影响“程序能否编译通过”。

- `timeout_ms`：请求下游模型接口的超时上限（毫秒），你的理解正确。

- `mutex` 写锁：你的理解正确，本质是“同一时刻只允许一个写入者”，避免多个 POST 同时写 `llm.json` 造成覆盖混乱。  
  类比：一个公共账本只有一把钥匙，拿到钥匙的人先写，写完再交给下一个人。

## 关键流程/步骤（可复刻）

### GET /api/v1/settings

1. 读取当前 LLM 配置（可走缓存）。
2. 对 API Key 做掩码（只显示后 4 位）。
3. 返回可展示字段给前端。

### POST /api/v1/settings

1. 解析 JSON body（支持部分字段）。
2. 做字段校验（如 `temperature`、`timeout_ms` 范围）。
3. 进入 `mutex` 临界区（加锁）。
4. 读取 `config/llm.json` 旧配置。
5. 合并新字段后写回文件。
6. 立即调用缓存失效函数，让新配置马上生效。
7. 返回 `{"status":"ok","message":"settings updated"}`。

### temperature 何时生效（时间线）

1. 你在设置页保存（POST settings）或手改 `llm.json`。  
2. 后端更新配置并刷新缓存。  
3. 下一次用户调用 `/plan` 或 `/chat/stream`。  
4. 后端向下游模型发送请求时带上 `temperature`。  
5. 这一次模型回复就按新 `temperature` 风格生成。  

## 真实代码片段（逐行注释）

```cpp
std::mutex g_settings_file_mutex;                              // 全局互斥锁：保护配置文件写入

Json::Value handle_get_settings() {                            // GET /api/v1/settings 处理函数
    auto cfg = load_llm_config_cached();                       // 读取当前配置（可使用缓存）
    Json::Value out;                                            // 构造返回 JSON
    out["api_base"] = cfg.api_base;                            // 返回模型服务地址
    out["model"] = cfg.model;                                  // 返回模型名称
    out["temperature"] = cfg.temperature;                      // 返回温度参数
    out["timeout_ms"] = cfg.timeout_ms;                        // 返回超时参数
    out["api_key_configured"] = !cfg.api_key.empty();          // 只告诉前端是否已配置 API Key
    out["api_key_hint"] = mask_last4(cfg.api_key);             // 只暴露掩码后的后 4 位
    return out;                                                 // 返回给前端
}                                                               // GET 结束

Json::Value handle_post_settings(const Json::Value& body) {    // POST /api/v1/settings 处理函数
    std::lock_guard<std::mutex> lock(g_settings_file_mutex);   // 进入临界区：自动加锁
    auto old_cfg = read_json("config/llm.json");               // 读取旧配置
    auto new_cfg = merge_partial(old_cfg, body);               // 合并“只传入”的字段
    write_json("config/llm.json", new_cfg);                    // 写回配置文件
    invalidate_llm_config_cache();                             // 立即失效缓存，确保马上生效
    return Json::parse(R"({"status":"ok","message":"settings updated"})"); // 返回成功结果
}                                                               // 作用域结束：lock_guard 自动解锁
```

## 小结 + 自测题（含答案）

小结：
- 你对 GET/POST、`timeout_ms`、`mutex 防并发写冲突` 的理解是正确的。
- Phase 6 的关键是：先把“可改设置 + 立即生效 + 基础测试 + Release 性能”打通。
- `temperature` 不是“越大越好”，而是按业务稳定性与创造性需求调。

自测题：
1. 为什么 POST settings 写文件要加 `mutex`？
2. `temperature=0.2` 和 `temperature=1.0` 的输出风格差异是什么？
3. 为什么 POST 成功后要立刻 `invalidate_llm_config_cache()`？
4. `Release` 和 `Debug` 的核心差别是什么？

答案：
1. 防止并发写入导致配置覆盖、文件内容错乱或最后一次写入不可预期。
2. `0.2` 更稳定保守，`1.0` 更发散灵活。
3. 否则会被 5 秒 TTL 缓存挡住，新配置不会“立刻”生效。
4. Debug 便于调试；Release 开启优化，性能更高，适合交付运行。

## 下一步建议

1. 在实现 POST 时补充参数校验：`temperature`（如 0~2）、`timeout_ms`（>0）。
2. 用 gtest 先补 3 个关键用例：GET 掩码正确、POST 部分更新正确、POST 后缓存立即失效。
