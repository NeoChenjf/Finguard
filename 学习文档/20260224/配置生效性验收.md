## 核心概念

### 什么是"配置生效性"？

> 验收问题：我改了 `config/rate_limit.yaml`，系统会不会真的按新值运行？

"配置生效性"验收的是**三件事**：
1. 配置文件能被成功**读取**（文件存在、格式正确）。
2. 读取到的值被**正确使用**（代码实际用了这个值去判断）。
3. 修改配置后**多快能生效**（需要重启？还是立刻生效？）。

---

## 项目配置文件一览

项目把配置分成 5 个独立文件，每个管一块：

| 文件 | 管什么 |
|------|--------|
| `config/rate_limit.yaml` | 入口限流速率/容量、模型桶速率/容量 |
| `config/concurrency.yaml` | 最大同时处理请求数（`max_inflight`） |
| `config/timeout.yaml` | 各路由/模型的超时时间与重试次数 |
| `config/circuit_breaker.yaml` | 熔断器的失败率阈值、窗口、半开探测 |
| `config/observability.yaml` | 日志字段、指标开关 |

每个文件都有**内置默认值**：如果文件不存在或解析失败，代码会回退到安全的默认配置，不会崩溃。

## 关键机制：热更新（每次请求都重新读文件）

这是本项目配置加载最重要的特性：

```cpp
// routes.cpp — /api/v1/plan 路由处理器内部

app().registerHandler("/api/v1/plan", [](const HttpRequestPtr &req, ...) {
    // ↓ 注意：load_rate_limit_config() 在请求处理函数内部被调用
    const auto rate_cfg = util::load_rate_limit_config();   // 每次请求都重新读文件
    const auto conc_cfg = util::load_concurrency_config();  // 每次请求都重新读文件

    g_concurrency_limiter.set_max_inflight(conc_cfg.max_inflight); // 立刻更新并发上限
    // ...后续用 rate_cfg 里的值去判断限流
});
```

**白话**：`load_rate_limit_config()` 不是在程序启动时调用一次，而是**每来一个请求就调用一次**，每次都打开文件重新读一遍。

### 这意味着什么？

```
你修改 config/rate_limit.yaml
         │
         ↓（不需要重启服务器）
下一个请求进来
         │
         ↓ load_rate_limit_config() 重新读文件
         ↓ 拿到新的 rate_rps / capacity
新配置立刻生效 ✅
```

> 类比：餐厅每位服务员接单时都去看一眼今天的菜单，而不是早上背一次就用一整天。换菜单不需要重新开门营业。

## 配置加载的防御机制（不会因配置错误崩溃）

```cpp
// reliability_config.cpp — load_rate_limit_config() 内部

RateLimitConfig load_rate_limit_config() {
    RateLimitConfig cfg;
    // 先设好默认值（qwen: rate=2, capacity=6 等）
    cfg.model_limits["qwen"] = {2.0, 6};
    cfg.model_limits["deepseek"] = {1.0, 3};

    const auto path = ... / "config" / "rate_limit.yaml";
    if (!std::filesystem::exists(path)) {
        return cfg;  // 文件不存在 → 直接用默认值，不报错
    }

    if (!parse_simple_yaml(load_file(path), doc, &error)) {
        return cfg;  // 解析失败 → 直接用默认值，不报错
    }

    // 只有解析成功，才用文件里的值覆盖默认值
    if (doc.scalars.count("entry.rate_rps")) {
        cfg.entry.rate_rps = std::stod(doc.scalars["entry.rate_rps"]);
    }
    // ...
    return cfg;
}
```

**三层保护**：
1. 文件不存在 → 用默认值
2. 文件格式有误 → 用默认值
3. 某个字段缺失 → 只有存在的字段才覆盖，其余保留默认

## 验收证据：`/metrics` 和 `/health`

### `/health`——配置能读取的间接证据

```bash
GET http://localhost:8080/health
# 返回：{"status":"ok"}
```

服务器能正常响应说明启动时配置加载没有导致崩溃。

### `/metrics`——配置真正被使用的直接证据

```bash
GET http://localhost:8080/metrics
# 返回：
{
  "requests_total": 42,
  "rate_limit_rejects_total": 7,    ← 这个数字 > 0 说明限流配置在工作
  "circuit_breaker_trips_total": 1,
  "latency_p95_ms": 312.4,
  "latency_p99_ms": 891.0,
  "external_call_latency_ms_p95": 650.2
}
```

`rate_limit_rejects_total > 0` 说明：
1. 限流配置被读取了（否则始终放行，这个数永远是 0）。
2. 读取到的 `rate_rps` / `capacity` 值确实被用于 `g_entry_bucket.allow()` 的判断。

## 小结

- **配置生效性**验收的是：改了配置文件，系统是否真的按新值运行。
- 本项目采用**每次请求重新读文件**的方式，修改 YAML 后**无需重启**，下一个请求立刻生效（真热更新）。
- 读取逻辑有**三层防御**，文件缺失或格式错误不会崩溃，回退默认值。
- 验收证据：`/health` 证明启动没崩，`/metrics` 的计数器增长证明配置值被真正使用。

## 自测题（含答案）

**1) 本项目的配置热更新是怎么实现的？**
> 答案：`load_rate_limit_config()` 等函数被放在**请求处理函数内部**调用，每来一个请求就重新打开并解析一次 YAML 文件，所以文件改动后下一个请求立刻生效，无需重启。

**2) 如果 `config/rate_limit.yaml` 文件写错了（YAML 格式错误），服务器会崩溃吗？**
> 答案：不会。解析失败时代码直接 `return cfg`，使用内置默认值继续运行。

**3) `rate_limit_rejects_total` 这个指标为什么能证明配置生效？**
> 答案：它是在 `g_entry_bucket.allow()` 返回 `false` 时才 +1 的。`allow()` 内部用的正是从配置文件读来的 `rate_rps` 和 `capacity`。如果配置没生效（始终用 0 或极大值），这个计数不会增加。

**4) 5 个配置文件各自互不影响，这样设计有什么好处？**
> 答案：改限流配置不会误碰超时配置；某个文件解析失败只影响那一块功能，其他功能正常运行。方便分模块调试和独立热更新。
