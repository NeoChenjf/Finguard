# 压测入门与流式/非流式实践（Phase5）

## 标题与目标

目标：用最小概念成本理解“非流式/流式压测”、`hey`、`k6`、以及“并发模型从 Start-Job 改为 asyncio/aiohttp”的原因，并能写出可复刻的压测命令与脚本骨架。

## 学习路线/分层

1) 先分清：非流式 vs 流式（只看概念）
2) 再认识工具：`hey` 与 `k6`（知道为何 Phase5 选择/不选择）
3) 再理解并发模型：Start-Job vs asyncio/aiohttp（知道性能与可控性差异）
4) 最后把它们串成“压测流程”：准备 → 运行 → 记录 → 对比

## 核心概念解释（带生活类比）

- 非流式接口：一次请求 -> 一次完整响应。像“点菜→厨房做好→一次性上菜”。
- 流式接口：一次请求 -> 多次分块返回。像“点菜→厨房边做边上小菜→最后上主菜”。
- 压测：用很多人同时点菜，观察厨房能不能扛得住、速度有没有掉。
- `hey`：一个简单快速的“集体点菜”工具，适合非流式接口。
- `k6`：更复杂、更强大的工具，但引入成本更高；Phase5 选择先不用。 (来自 phase5.md)
- 并发模型：
  - Start-Job（PowerShell 后台任务）：像请很多“临时服务员”各自忙，管理成本高。
  - asyncio/aiohttp（Python 异步）：像一个“总管”同时盯很多订单，效率更高。

## 关键流程/步骤（可复刻）

1) 明确场景与端点
   - 非流式：如 `/health`、`/api/v1/plan`
   - 流式：如 `/api/v1/chat/stream`

2) 选择工具
   - 非流式：用 `hey`
   - 流式：用 Python SSE 脚本（Phase5 建议改为 asyncio/aiohttp） (来自 phase5.md)

3) 运行压测并记录结果
   - 记录响应时间（P95/P99）、吞吐量（RPS）、错误率
   - 结果按 Phase5 约定保存到 `tools/temp/`

4) 对比与结论
   - 同一场景重复跑两次，比较是否稳定
   - 优化后要回归压测

## 真实命令/代码示例（带逐行注释）

### 1) 非流式：hey 压 `/health`

```powershell
# 进入项目目录
Set-Location d:\AI_Investment
# 使用 hey 压测 /health，100 并发，持续 30 秒
hey -n 0 -c 100 -z 30s http://127.0.0.1:8000/health
```

说明：
- `-c 100` 表示并发 100。
- `-z 30s` 表示持续 30 秒。
- `-n 0` 表示不限制总请求数（以时间为准）。

### 2) 流式：Python asyncio/aiohttp 骨架（替代 Start-Job）

```python
import asyncio  # 异步调度器
import aiohttp  # 异步 HTTP 客户端

URL = "http://127.0.0.1:8000/api/v1/chat/stream"  # 流式接口地址
CONCURRENCY = 10  # 并发数

async def one_stream(session: aiohttp.ClientSession) -> None:
    # 发起一次流式请求
    async with session.get(URL) as resp:
        # 逐行读取 SSE 数据
        async for line in resp.content:
            _ = line  # 这里仅示例：真实脚本可统计首 token 延迟

async def main() -> None:
    # 复用连接，提高效率
    async with aiohttp.ClientSession() as session:
        # 同时启动多个请求任务
        tasks = [asyncio.create_task(one_stream(session)) for _ in range(CONCURRENCY)]
        # 等待所有任务结束
        await asyncio.gather(*tasks)

if __name__ == "__main__":
    # 运行异步主函数
    asyncio.run(main())
```

为什么比 Start-Job 更好：
- Start-Job 需要创建大量后台进程/任务，管理成本高。
- asyncio/aiohttp 用单进程多协程，开销小、并发控制更精细。 (来自 phase5.md)

### 3) 为什么 Phase5 不用 k6（只需知道原因）

- k6 很强，但工具链更复杂、学习成本更高。
- Phase5 已有 `hey` + Python SSE 脚本可满足需求。 (来自 phase5.md)

## 小结 + 自测问题（含答案）

小结：
- 非流式是“一次性返回”，流式是“分块返回”。
- `hey` 适合非流式压测，Python SSE 脚本适合流式压测。
- Start-Job 并发成本高，asyncio/aiohttp 更轻量。
- Phase5 明确不引入 k6，以降低复杂度。

自测题：
1) 为什么流式接口不用 `hey`？
2) `-c 100 -z 30s` 各代表什么？
3) asyncio/aiohttp 相比 Start-Job 的优势是什么？

答案：
1) `hey` 更适合一次请求一次响应，流式需要持续读取分块数据，Python SSE 脚本更合适。
2) `-c 100` 是并发 100，`-z 30s` 是持续 30 秒。
3) asyncio/aiohttp 采用单进程多协程，开销更低、并发更可控。
