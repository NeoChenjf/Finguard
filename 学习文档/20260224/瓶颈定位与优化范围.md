# 瓶颈定位与优化范围入门（Phase5）

## 标题与目标

目标：用小白能懂的方式解释“无数据不优化、Top 3 热点、CPU 热点、JSON 序列化/反序列化、正则匹配、锁竞争、共享状态、内存分配、flush”等概念，并知道它们在性能优化里的位置。

## 学习路线/分层

1) 先理解“瓶颈定位”与“热”是什么
2) 再理解“无数据不优化”与 Top 3 热点原则
3) 再逐个解释常见热点类型（CPU/JSON/正则/锁/内存/flush）
4) 最后把概念串起来：采集 → 定位 → 优化 → 回归

## 核心概念解释（带生活类比）

- 无数据不优化：像修车先看“行车记录”而不是凭感觉乱换零件。
- Top 3 热点：只先修“最堵的三条路”，优先解决影响最大的瓶颈。
- CPU 热点：最耗 CPU 的函数/代码段，像“最费火力的炉灶”。
- JSON 序列化/反序列化：把对象变成文本、再从文本还原，像“写菜单/读菜单”。
- 正则匹配：用规则找文本，像“在一大堆订单里用复杂条件找某几条”。
- 锁竞争：多人抢一把钥匙，谁拿到谁能继续干活，其他人只能等。
- 共享状态（token_bucket / metrics_registry / circuit_breaker）：多个线程/请求共用的“公共账本/开关/限流器”。
- 内存分配：申请/释放内存，像“不断拿新盘子、再回收盘子”。
- flush：把缓存的数据“立刻写出去”，像“把积攒的菜一次性端出去”。

## 关键流程/步骤（可复刻）

1) 先采集数据（Profiler / 压测结果）
2) 找出 Top 3 热点
3) 判断热点类型（CPU/锁/内存/IO）
4) 优化一个点后回归压测

## 小结 + 自测问题（含答案）

小结：
- “无数据不优化”是性能优化的铁律。
- 先抓 Top 3 热点，避免到处乱改。
- 常见热点来自 CPU、JSON、正则、锁竞争、内存分配、flush。
- 共享状态容易引发锁竞争与性能波动。

自测题：
1) 为什么要先找 Top 3 热点？
2) JSON 序列化/反序列化是什么？
3) 什么是锁竞争？
4) 共享状态为什么可能导致性能下降？

答案：
1) 先解决影响最大的瓶颈，效率最高。
2) 把对象转成 JSON 文本，再从文本还原。
3) 多个线程抢同一把锁，只有一个能先通过。
4) 多个线程同时读写共享对象，容易互相等待或加锁。

## 下一步建议

1) 用 Profiler 跑一次 CPU 热点清单。
2) 标注是否存在锁竞争或频繁内存分配。
3) 只选一个热点优化并回归压测。
