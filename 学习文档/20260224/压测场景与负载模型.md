# 压测场景与负载模型入门（Phase5）

## 标题与目标

目标：用小白能懂的方式解释“基线（空载）、并发梯度、mock、chunk、混合场景、稳定性场景、chat 起点/低起点/拐点、plan 与 chat、内存泄漏”等概念，并能看懂 S1-S5 场景表。

## 学习路线/分层

1) 先理解“场景”和“负载模型”是什么
2) 再逐个解释关键术语（基线、并发梯度、mock、chunk…）
3) 最后把术语放回 S1-S5 场景里理解

## 核心概念解释（带生活类比）

- 场景：就是“模拟一种真实使用方式”。像餐厅有“午高峰、晚高峰、深夜单人点单”。
- 负载模型：就是“给厨房上多少单、怎么慢慢加单”。
- 基线（空载）：像餐厅还没来客人时，看看厨房本身的速度与状态。
- 并发梯度：像每隔一段时间加一波客人（50 → 100 → 200 → 500）。
- mock：用“假下游”代替真实下游。像用“假厨房”模拟做菜速度，避免真实厨房波动。
- chunk（流式分块）：流式接口不是一次性端上菜，而是分多次上小菜（每次一小碟）。
- 混合场景：同时有“点菜”和“边上小菜”的订单，测试资源是否互相抢。
- 稳定性场景：长时间开店，看是否“越做越慢或漏水”。
- chat 起点/低起点：流式连接贵，先从少量并发开始试探。
- 拐点：并发增加到某个点时，性能突然变差（延迟变大、错误率变高）。
- plan 与 chat：两个不同接口，一个是“算方案”，一个是“聊天流式输出”。
- 内存泄漏：像水桶有小洞，运行越久水越少，内存越跑越高。

## 关键流程/步骤（可复刻）

1) 先跑基线（S1）
   - 确认系统“空载”下的性能指标
2) 按并发梯度逐步加压
   - 观察延迟和错误率何时开始明显恶化
3) 跑混合场景（S4）
   - 验证 plan 和 chat 同时存在时是否互相影响
4) 跑稳定性场景（S5）
   - 看 10 分钟内内存是否持续上升

## 真实命令/代码示例（带逐行注释）

### 1) 基线场景（S1）示例：`/health`

```powershell
# 进入项目目录
Set-Location d:\AI_Investment
# 基线场景：用 hey 压 /health，50 并发，30 秒
hey -n 0 -c 50 -z 30s http://127.0.0.1:8000/health
```

说明：
- 这是“空载基线”，下游为“无”。
- 如果基线数据都不好，说明系统本身就有问题。

### 2) 并发梯度示例

```powershell
# 逐级增加并发（示例 50 → 100 → 200）
hey -n 0 -c 50  -z 30s http://127.0.0.1:8000/health
hey -n 0 -c 100 -z 30s http://127.0.0.1:8000/health
hey -n 0 -c 200 -z 30s http://127.0.0.1:8000/health
```

说明：
- “并发梯度”就是按阶梯式加压。
- 这样更容易发现“拐点”。

## 小结 + 自测问题（含答案）

小结：
- 基线（空载）是“系统最轻松时的表现”。
- 并发梯度是“逐步加压”而不是一步到位。
- mock 是“假下游”，chunk 是“流式分块返回”。
- 混合场景用于验证不同接口的资源竞争。
- 稳定性场景是长时间观察内存与性能。
- chat 起点/低起点用于更准确发现拐点。

自测题：
1) 为什么要先跑基线（空载）？
2) mock 和 chunk 分别是什么？
3) 混合场景主要验证什么？
4) 稳定性场景主要看什么指标？

答案：
1) 先知道系统在“最轻松时”的表现，作为对照基准。
2) mock 是假下游，chunk 是流式接口的分块返回。
3) 验证 plan 与 chat 同时存在时是否互相抢资源、导致退化。
4) 重点看内存是否持续增长（泄漏）与性能是否持续下降。
