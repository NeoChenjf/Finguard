## 核心概念：令牌桶（Token Bucket）

### 白话解释

令牌桶是一种限流算法。概念很简单：

> 有一个桶，里面放着"令牌"（token）。每次请求来了，拿走 1 个令牌才能通过；没令牌就被拒绝。桶会按固定速率自动补充令牌，但总量不超过桶的容量上限。

### 生活类比

想象一个停车场入口的**取票机**：

- 取票机里有一叠票（令牌），最多放 15 张（`capacity`）。
- 每秒自动补充 5 张新票（`rate_rps = 5`）。
- 每辆车进场取走 1 张票。
- 票用完了？后面的车直接被拦住，等票补充。

关键结论：
- **`rate_rps`（速率）**：每秒补充多少令牌 = 稳定状态下每秒允许多少请求通过。
- **`capacity`（容量）**：桶最多存多少令牌 = 允许短暂突发多少请求（比如瞬间来了 6 个，只要桶里有 6 个令牌，就全部放行）。

## 什么是"模型桶"？

**模型桶**就是一个专门针对"调用 LLM 模型"这件事的令牌桶。

### 为什么要专门建一个？

项目里有两层限流：

| 层级 | 限的是 | 对应的桶 |
|------|--------|----------|
| 入口层 | 外部用户请求（user_id + route） | `g_entry_bucket` |
| 模型层 | 对 LLM API 的调用（model family） | `model_bucket` |

入口限流防止**用户把系统打爆**；模型桶防止**系统把 LLM API 调用费用打爆**或**超出 LLM 平台的 QPS 限制**。

### 为什么按"族名"分组而不是具体型号？

配置里是这样写的：

```yaml
# config/rate_limit.yaml
model:
  qwen:         # "qwen" 包含 qwen-plus、qwen-turbo、qwen-max 等所有 qwen 系列
    rate_rps: 2
    capacity: 6
  deepseek:     # "deepseek" 包含 deepseek-chat、deepseek-coder 等
    rate_rps: 1
    capacity: 3
```

原因：如果按具体版本号分桶（比如 `qwen-plus`、`qwen-turbo` 各自一个桶），那每次模型版本升级就要改配置。按族名（family）归类，一组共用一个桶，维护成本低，还能统一控制整族调用成本。

## 项目里的完整流程

```
一个用户请求到达
       │
       ▼
[入口层令牌桶] g_entry_bucket.allow("user:xxx + /api/v1/chat")
       │ 没令牌 → 立刻 429 返回给用户
       │ 有令牌 ↓
       ▼
[解析模型族名] model_family("qwen-plus") → "qwen"
       │
       ▼
[模型桶令牌桶] model_bucket.allow("model:qwen", rate_rps=2, capacity=6)
       │ 没令牌 → 降级响应（不调用 LLM，返回 fallback）
       │ 有令牌 ↓
       ▼
[调用 LLM API]（真正发出 HTTP 请求）
```

## 真实代码（逐行注释）

### 1. 族名提取函数（llm_client.cpp）

```cpp
// 把具体模型名映射成族名
std::string model_family(const std::string &model) {
    std::string out = model;
    // 全部转为小写，方便比较（qwen-PLUS → qwen-plus）
    for (auto &c : out) {
        c = static_cast<char>(std::tolower(static_cast<unsigned char>(c)));
    }
    // 只要模型名里包含 "qwen"，就归类为 qwen 族
    if (out.find("qwen") != std::string::npos) {
        return "qwen";
    }
    // 包含 "deepseek" 归类为 deepseek 族
    if (out.find("deepseek") != std::string::npos) {
        return "deepseek";
    }
    // 其他未知模型用 default 族（避免崩溃）
    return "default";
}
```

### 2. 模型桶检查（llm_client.cpp 调用入口）

```cpp
// 读取限流配置（从 config/rate_limit.yaml）
const auto rate_cfg = util::load_rate_limit_config();

// 定义模型桶（static = 只初始化一次，贯穿整个程序生命周期）
static util::TokenBucket model_bucket;

// 查找当前模型族对应的限流配置
const auto model_it = rate_cfg.model_limits.find(family); // family = "qwen"
const auto model_limit = (model_it != rate_cfg.model_limits.end())
    ? model_it->second          // 找到了 → 用配置里的 qwen 规则
    : util::RateLimitModelConfig{};  // 没找到 → 用默认值（空=不限流）

// 向模型桶申请令牌
if (!model_bucket.allow("model:" + family, model_limit.rate_rps, model_limit.capacity)) {
    // 申请失败 → 没令牌 → 降级，不调用 LLM
    util::global_metrics().record_rate_limit_reject(); // 记录一次拒绝指标
    result.degraded = true;                             // 标记为降级响应
    result.error = "model_rate_limited";                // 错误原因
    result.full_text = "FinGuard fallback response due to model rate limiting.";
    return result; // 直接返回，完全不发请求给 LLM
}
// 申请成功 → 继续往下调用 LLM API
```

### 3. 令牌桶的补充逻辑（token_bucket.cpp）

```cpp
bool TokenBucket::allow(const std::string &key, double rate_rps, int capacity) {
    // rate_rps <= 0 或 capacity <= 0 视为不限流，直接放行
    if (rate_rps <= 0.0 || capacity <= 0) {
        return true;
    }

    const auto now = std::chrono::steady_clock::now(); // 当前时间
    std::lock_guard<std::mutex> lock(mutex_);           // 加锁（线程安全）

    // 找到或新建这个 key 的桶状态
    auto [it, inserted] = buckets_.try_emplace(key);
    auto &state = it->second;
    if (inserted) {
        // 桶第一次创建：初始化为满桶
        state.tokens = static_cast<double>(capacity);  // 初始令牌 = 桶容量
        state.last   = now;
    }

    // 计算上次到现在经过了多少秒
    const double elapsed = std::chrono::duration<double>(now - state.last).count();
    // 按速率补充令牌，但不超过桶的容量上限
    state.tokens = std::min<double>(capacity, state.tokens + elapsed * rate_rps);
    state.last = now; // 更新时间戳

    // 桶里有令牌（>= 1）就扣掉 1 个，放行
    if (state.tokens >= 1.0) {
        state.tokens -= 1.0;
        return true;  // 放行
    }
    return false; // 没令牌，拒绝
}
```

## 用数字感受一下（qwen 桶的例子）

配置：`rate_rps: 2`，`capacity: 6`

```
t=0s  桶初始化：tokens = 6（满桶）
      来了 3 个请求 → 扣 3 个令牌 → tokens = 3，3 个请求放行

t=0.1s 来了 10 个请求
       先补充令牌：0.1s * 2 rps = 0.2 个 → tokens = 3.2
       放行 3 个（每次扣 1）→ tokens = 0.2，剩余 7 个被拒绝

t=1s  又经过了 0.9s，补充：0.9 * 2 = 1.8 → tokens = 2.0
      来了 2 个请求 → 全部放行 → tokens = 0
```

**结论**：稳定状态下每秒最多放行 2 个（rate_rps），偶发高峰可以借桶里最多 6 个令牌（capacity）撑一下。

## 小结

- **令牌桶**：一种限流算法。桶满足突发，速率控平均，没令牌就拒绝。
- **模型桶**：专门对 LLM 调用施加的令牌桶，防止调用频率和费用失控。
- **按族名分桶**：qwen 所有版本共用一个桶，降低维护成本，统一管控外部成本。
- **key = `"model:" + family`**：每个族各自独立一个桶，互不影响（qwen 满了不影响 deepseek）。

## 自测题（含答案）

**1) rate_rps 和 capacity 各控制什么？**
> 答案：`rate_rps` 控制**持续速率**（每秒补充/允许多少）；`capacity` 控制**突发上限**（瞬间最多能通过多少）。

**2) 如果模型桶没有令牌，请求会怎样？是 429 吗？**
> 答案：不是 429。模型桶耗尽走**降级**（`result.degraded = true`），返回一段 fallback 文字，而不是报错给用户。429 只在入口层出现。

**3) 为什么 TokenBucket 用 `static` 关键字定义？**
> 答案：`static` 让桶的状态（tokens、last）在多次函数调用之间持久保留，否则每次函数调用都会新建一个满桶，限流就永远不会触发了。
>
> **展开解释**：
>
> `static` 修饰函数内部局部变量时，含义是：**这个变量只在程序启动后初始化一次，之后每次调用该函数都继续使用上次留下的值，不会重置。**
>
> 对比两种情况：
>
> | | 没有 `static` | 有 `static` |
> |---|---|---|
> | 每次请求进来 | 新建一个满桶（tokens = 6） | 沿用上次的桶状态 |
> | 扣令牌的效果 | 下次请求又是满桶，等于没扣 | 令牌被真正累计扣减 |
> | 限流是否生效 | ❌ 永远放行 | ✅ 正常限流 |
>
> 代码里的具体位置：
> ```cpp
> // 这个函数每次收到请求都会被调用
> LlmResult LlmClient::call(...) {
>     // static：只在第一次调用时创建，之后一直复用
>     static util::TokenBucket model_bucket;
>     //        ^^^^^^
>     //        去掉 static → 每次进来都是新桶 → 限流失效
>
>     model_bucket.allow("model:qwen", 2.0, 6); // 在同一个桶上扣令牌
> }
> ```
>
> 生活类比：取票机（`static`）vs 每次进门发一张全新的票单（无 `static`）。前者票会被真正取走；后者每个人进门都领到一张全新的票单，永远不会没票。

**4) qwen-plus 和 qwen-turbo 会共享同一个桶吗？**
> 答案：会。`model_family("qwen-plus")` 和 `model_family("qwen-turbo")` 都返回 `"qwen"`，key 都是 `"model:qwen"`，共用同一个桶状态。
