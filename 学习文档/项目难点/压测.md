## 核心概念：为什么会内存泄漏？

### 白话类比：食堂的一个故障

想象一个食堂每次有顾客来就这样做：

**优化前的做法（错的）**：
```
顾客1来 → 查一遍菜单（打开大厚菜单） → 点菜 → 吃饭 → 关菜单
顾客2来 → 查一遍菜单（又打开一遍）  → 点菜 → 吃饭 → 关菜单
顾客3来 → 查一遍菜单（又打开一遍）  → 点菜 → 吃饭 → 关菜单
... (180个顾客 = 180次打开菜单)
```

每打开一次菜单，都要：
- 从柜子里拿出来（IO成本）
- 把每道菜的内容读一遍、理解一遍（解析成本）
- 放回去（销毁成本）

**问题**：180次打开 = 180次"拿出、理解、放回" = 大量内存抖动 = GC跟不上 = 内存积压

**优化后的做法（对的）**：
```
前5分钟里：顾客1,2,3...都看同一份菜单（放在柜台上）
  ↓
满5分钟 → 再查一次，确认菜单有没有改
  ↓
接下来5分钟里：顾客又看同一份菜单
```

结果：从"每顾客查一次"变成"每5分钟查一次"

## 真实现场：问题发现

### 压测场景与基线数据

运行 10 分钟压测（S5 稳定性测试）：
```
并发数：20 个 SSE 长连接
吞吐：180 RPS
端点：/api/v1/chat/stream（调用下游 LLM）
```

|------|--------|------|
| 起始 RSS | 55.92 MB | 程序启动时内存 |
| 结束 RSS | 88.21 MB | 10分钟后内存 |
| RSS 增长 | **+57.76%** | 内存泄漏标记 ⚠️ |
| P95 延迟 | 123.24 ms | 随时间推移还在增加 |

```
时间线：
0 min   → RSS: 56 MB,   P95: 110ms
2 min   → RSS: 64 MB,   P95: 115ms
5 min   → RSS: 72 MB,   P95: 120ms
10 min  → RSS: 88 MB,   P95: 123ms

特征：线性增长，没有下降（真泄漏，不是暂时抖动）
```

## 关键流程：从假设到真相

### 第一阶段：错误的假设（磁盘 I/O）

**我的初始诊断**：
```
根据 phase 5 章节 3.3，Top 3 瓶颈：
1. 每请求加载配置文件（磁盘 I/O）← 180 RPS = 900+ 次I/O/秒
2. ...
```

**为什么这个假设"看起来对"？**
- 代码确实在每次请求时调用了 `load_rate_limit_config()` 
- 代码确实在打开文件
- ✅ 表面看没问题

**但用户的质疑推翻了这个假设**：

> 你认为这个后端的架构不至于蠢到每次都从磁盘里面读吧？

**真相**：操作系统会把常访问的文件缓存在内存里。  
再频繁的 `fopen()` 调用也不会真正去磁盘 I/O，而是返回 OS 的页缓存。

❌ **所以"磁盘 I/O"不是瓶颈**

### 第二阶段：真正的瓶颈（YAML 解析）

在用户的质疑下，我重新看代码：

```cpp
RateLimitConfig load_rate_limit_config() {
    RateLimitConfig cfg;
    
    // ✅ 这一步可能被OS缓存，成本低
    std::string content = load_file(path);
    
    // ❌ 这一步才是真正的成本！
    YamlDoc doc;
    parse_simple_yaml(content, doc, &error);  // 创建临时对象
    
    // ❌ 这一步也很贵
    if (doc.scalars.count("entry.rate_rps")) {
        cfg.entry.rate_rps = std::stod(doc.scalars["entry.rate_rps"]);
    }
    // ... 还有10多个 stod/stoi + unordered_map 操作
    
    return cfg;  // 销毁 doc 及所有临时 string
}
```

**真正的成本**：
- 每次调用 = 创建 YamlDoc 对象 + string 拷贝 + 类型转换 + unordered_map 插入
- 180 RPS = **每秒 180 次上述操作**
- 频繁的 malloc/free = 堆碎片化 = GC 无法及时回收整页内存 = RSS 线性增长

---

## 解决方案设计

### 为什么不直接"优化 YAML 解析算法"？

❌ **思路 1：加快 stod 和 unordered_map 的速度**
- 复杂度：高（改写 YAML 库）
- 收效：有限（算法再快，一秒 180 次调用也是累赘）
- 风险：容易引入bug

✅ **思路 2：减少解析的频率**
- 复杂度：低（加个缓存模板）
- 收效：巨大（180 次/秒 → 0.2 次/秒）
- 风险：低（缓存逻辑简单）

**结论**：选思路 2

### 实现：ConfigCache<T> 模板

核心思路：
```cpp
template <typename T>
struct ConfigCache {
    std::mutex mtx;
    T value{};           // 缓存值，保存一份
    Clock::time_point loaded_at{};

    T get(Loader loader) {
        std::lock_guard<std::mutex> lock(mtx);
        const auto now = Clock::now();
        
        // 只有过期了（超过5秒）才重新加载
        if (now - loaded_at > std::chrono::seconds(5)) {
            value = loader();       // 调用一次 load_rate_limit_config()
            loaded_at = now;
        }
        
        return value;  // 返回这一份缓存的值
    }
};
```

**关键设计**：
- **5 秒 TTL**：配置文件这么"稳定"的东西，5 秒刷新一次足够了
- **全局单例**：用全局变量 `ConfigCache<RateLimitConfig> g_rate_limit_cache`，所有请求共享
- **线程安全**：用 `std::mutex` 保护，避免并发冲突
- **懒加载**：第一次调用时才真正加载，后续都返回缓存

### 替换调用点

**改前**：
```cpp
// routes.cpp
RateLimitConfig cfg = load_rate_limit_config();  // 每次都调用
```

**改后**：
```cpp
// routes.cpp
RateLimitConfig cfg = cached_rate_limit_config();  // 用缓存版本
```

就这么简单。

## 关键代码说明

### ConfigCache 完整实现

**reliability_config.h**：
```cpp
// ── P5 perf: cached versions with 5-second TTL (thread-safe) ──
RateLimitConfig cached_rate_limit_config();       // ← 新增
TimeoutConfig cached_timeout_config();             // ← 新增
CircuitBreakerConfig cached_circuit_breaker_config(); // ← 新增
ConcurrencyConfig cached_concurrency_config();     // ← 新增
```

**reliability_config.cpp**：
```cpp
namespace {
    using Clock = std::chrono::steady_clock;
    constexpr auto kCacheTtl = std::chrono::seconds(5);  // 5秒过期
    
    // 缓存模板
    template <typename T>
    struct ConfigCache {
        std::mutex mtx;
        T value{};
        Clock::time_point loaded_at{};
        
        template <typename Loader>
        T get(Loader loader) {
            std::lock_guard<std::mutex> lock(mtx);
            const auto now = Clock::now();
            
            // 检查是否过期
            if (now - loaded_at > kCacheTtl) {
                value = loader();      // ← 真正的解析只在这里发生
                loaded_at = now;
            }
            return value;
        }
    };
    
    // 4 个全局缓存实例
    ConfigCache<RateLimitConfig> g_rate_limit_cache;
    ConfigCache<TimeoutConfig> g_timeout_cache;
    ConfigCache<CircuitBreakerConfig> g_cb_cache;
    ConfigCache<ConcurrencyConfig> g_conc_cache;
}

// 导出函数
RateLimitConfig cached_rate_limit_config() {
    return g_rate_limit_cache.get(load_rate_limit_config);
}
```

## 对比数据说话

### 优化前 vs 优化后

| 指标 | 优化前 | 优化后 | 改善 |
|------|--------|--------|------|
| **RPS (avg)** | 180 | 262 | +46% |
| **P50 延迟** | 111.10 ms | 68.16 ms | -39% |
| **P95 延迟** | 123.24 ms | 81.99 ms | -33% |
| **P99 延迟** | 132.59 ms | 98.78 ms | -26% |
| **RSS 起始** | 55.92 MB | 120.88 MB | — |
| **RSS 结束** | 88.21 MB | 120.77 MB | — |
| **RSS 增长%** | **+57.76% ❌** | **-0.09% ✅** | **内存泄漏消失** |

### 为什么 RSS 基础值变高了（120MB vs 56MB）？

因为缓存本身占用了内存（~65 MB）。

**但关键是**：虽然基础值升高，但**增长率从 +57.76% 变成 -0.09%**。

这意味着：
- 优化前：内存像漏水一样，每分钟漏 3MB
- 优化后：内存稳定，甚至略有回收（-0.09%）

## 小结

### 关键要点

1. **问题发现**：用压测工具的 RSS 增长曲线发现内存泄漏（不是靠猜）

2. **避免陷阱**：表面上看"磁盘 I/O 频繁"是瓶颈，但 OS 文件缓存会保护你。真正的瓶颈往往更深层。

3. **根因分析**：频繁的**配置解析**（不是 I/O），造成频繁的 malloc/free，导致堆碎片，最终 RSS 线性增长。

4. **优化策略**：与其优化解析算法（投入大、收效小），不如减少解析频率（投入小、收效大）。

5. **缓存设计**：用 5 秒 TTL 的缓存，让解析从 180 次/秒 降到 0.2 次/秒，问题迎刃而解。

## 自测题

### 题目 1：为什么 OS 文件缓存不能解决内存泄漏问题？

<details>
<summary>答案</summary>

因为泄漏不是来自"磁盘 I/O"，而是来自**每次 I/O 后的 YAML 解析**。

即使 `fopen()` 返回的是 OS 缓存中的文件内容，每次调用 `parse_simple_yaml()` 都会：
- 创建新的 YamlDoc 对象（malloc）
- 进行字符串操作和类型转换（malloc）
- 函数返回时销毁这些对象（free）

频繁的 malloc/free 导致堆碎片化，GC 无法及时回收整页内存，RSS 逐渐增长。

</details>

### 题目 2：为什么选择 5 秒而不是 1 秒或 10 秒的 TTL？

<details>
<summary>答案</summary>

这是权衡题：

- **TTL 太短**（如 1 秒）：缓存效果差，频繁解析，内存问题仍然存在
- **TTL 太长**（如 1 分钟）：配置文件改了，需要等待最多 1 分钟才能生效，不利于运维快速调试
- **5 秒**：平衡点
  - 对用户感知：5 秒延迟几乎无感（运维调试时可以接受）
  - 缓存效果：180 RPS 场景下，5 秒能处理 900 个请求，本地缓存的请求占 >99%
  - 内存稳定性：解析频率从 180/秒 → 0.2/秒，内存申请频率大幅下降

</details>

### 题目 3：代码中为什么用 `std::lock_guard<std::mutex>` 而不是直接访问缓存值？

<details>
<summary>答案</summary>

因为多个线程可能同时调用 `cached_rate_limit_config()`：

- **没有锁的情况**：线程 A 在检查"是否过期"时，线程 B 可能同时在修改 `value` 和 `loaded_at`，导致数据竞争，可能读到不一致的值。

- **用锁的情况**：保证同一时刻只有一个线程进入 `get()` 函数，修改或读取缓存都是原子的，避免脏读。

虽然每次都加锁看似增加开销，但：
- 缓存命中时锁的持有时间很短（几微秒）
- 相比 yaml 解析 (ms 级)，锁的开销可忽略不计

</details>

### 题目 4：如果把缓存时间改成 0（即永久不刷新），会怎样？

<details>
<summary>答案</summary>

程序运行期间，配置文件改了也不会生效。

例如运维想调整 `rate_rps` 从 5000 改到 3000，改好配置文件后，必须重启程序才能加载新配置。

这对线上服务很危险（无法灰度调整）。

所以即使解析成本很高，也需要定期刷新（通常 5-10 秒）来平衡安全性和性能。

</details>

## 下一步建议

### 进阶方向

1. **深入内存管理**
   - 学习堆碎片化的原理
   - 研究 malloc 的不同实现（glibc 的 ptmalloc vs jemalloc）
   - 实验：用不同 malloc 配置重跑压测，看 RSS 表现

2. **缓存更上层的设计**
   - 当前缓存的是"配置值"
   - 进阶：缓存"解析后的对象"（对象池模式）
   - 终极：异步后台刷新（避免锁争抢）

3. **运维集成**
   - 给配置加"版本号"，客户端检查版本而不是时间
   - 支持热更新（信号触发刷新，不需要重启）

4. **性能量化**
   - 用 CPU profiling 工具（如 perf/VTune）精确测量每个优化的收益
   - 建立性能 benchmark 套件，防止后续改动的回归
