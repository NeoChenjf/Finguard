# 概念解释（更详细生活例子 + 图解式文字）

## 1. 先想象你要寄一个包裹给朋友
你=程序；包裹=请求；朋友=LLM 服务。

### 场景 A：直连
你自己骑车送到朋友家。
- 好处：简单、路线短。
- 坏处：路上堵车、找不到路就失败。

### 场景 B：代理（中转站）
你把包裹交给中转站，由中转站再送到朋友家。
- 好处：中转站懂路线、能翻墙、能走专用通道。
- 坏处：要写清楚最终地址，不然中转站不知道往哪送。

### 场景 C：降级（备用路线）
如果你骑车送不通，就改坐地铁或打车。
- 目的：不一定最快，但保证能送到。

图解式文字（流程）

直连：
你 --> 朋友家

代理：
你 --> 中转站 --> 朋友家

降级：
你 --X--> 朋友家
  \--> 备用路线 --> 朋友家

---

## 2. 代理注入 / 降级
- 代理注入：
  - 类比：在包裹上贴“必须走中转站”的标签。
  - 系统看到这个标签，就把包裹先送去中转站。
- 降级：
  - 类比：高速封路了，就改走国道。
  - 目的：不追求速度，只求送达。

图解式文字：

主路成功：
请求 -> 直连 -> LLM

主路失败：
请求 -> 直连 X
             -> 降级(备用) -> LLM

---

## 3. http_proxy 字段
- 类比：写在纸条上的“中转站地址”。
- 没有这个字段：默认你自己送（直连）。
- 有这个字段：你把包裹交给中转站。

---

## 4. 连接代理 Host/Port + 绝对 URL + Host 头
### 生活类比
中转站员工问你：
- 你只说“送到 3 号楼 201”是不够的。
- 你必须说“北京市朝阳区某小区 3 号楼 201”。
- 同时在包裹上写“收件人名字”。

技术对应：
- Host/Port：中转站地址
- 绝对 URL：完整目的地地址
- Host 头：收件人名字（告诉对方你最终要找谁）

图解式文字：

你告诉中转站：
目标地址 = https://example.com/chat/completions
收件人 = example.com

---

## 5. curl fallback
- 类比：系统快递不通，你自己打电话叫车，把包裹送出去。
- 这个“电话叫车”就是 curl：独立、简单、能绕开系统里复杂的线路。

图解式文字：

系统线路（失败）：
程序 -> Drogon 请求 X

备用线路（成功）：
程序 -> curl 命令 -> LLM

---

## 6. SSE 流式（Server-Sent Events）
### 生活类比
- 想象你在看体育直播比分：不是等整场结束才告诉你结果，而是每进一球就推送一次。
- SSE 就是“服务器不断推送小更新”，客户端一直接收。

图解式文字：

普通响应（一次性）：
客户端 -> 请求 -> 服务器 -> 一次性完整结果

SSE（流式推送）：
客户端 -> 请求 -> 服务器 -> token1 -> token2 -> token3 -> done

关键点：
- 连接会保持更久，用来持续推送。
- 事件像“实时通知”，一次一小段。

---

## 7. proxy 透传
- 类比：备用司机也必须走同一个中转站。
- 你把“中转站地址”同时交给备用司机。
- 结果：主路/备用路都能通过代理。

---

## 8. 失败/降级路径
### 生活类比
- 主路堵车：记录“堵车原因”。
- 转备用路：继续送。
- 备用路也堵：记录“备用失败原因”，返回“临时方案”。

技术对应：
- 主路失败：ReqResult != Ok
- 进入降级：curl fallback
- 降级也失败：返回 mock/错误提示

---

## 9. ReqResult / HTTP status / timeout
### ReqResult（路上是否出事）
- DNS 找不到路
- TLS 握手失败
- 连接断了

### HTTP status（到达后对方回执）
- 200：收到并成功处理
- 401：你没权限
- 500：对方内部出错

### timeout（等太久）
- 你一直等朋友回短信，超过时间就放弃。

图解式文字：

请求发出 ->
  若路上出事 = ReqResult 失败
  若到达但被拒 = HTTP status 401/500
  若一直没回 = timeout

---

## 10. 小结（一句话记忆）
- 代理=中转站
- 降级=改走备用路线
- http_proxy=中转站地址
- Host 头=收件人名字
- ReqResult=路上事故
- HTTP status=签收回执
