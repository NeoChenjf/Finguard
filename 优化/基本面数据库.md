## Chapter 1：Spec pre-plan（冻结）

### 1) 新方案目标冻结：SQLite 成为 ValueCell v1 主数据源
- **决策/推荐**：ValueCell v1 由“请求期实时调用外部 API”切换为“请求期优先读取本地 SQLite 数据库”。
- **理由**：
  - 当前实时链路受上游接口稳定性、配额、响应时间影响明显；
  - ValueCell 主页面需要的是结构化、可重复、可复核的数据，而不是每次请求都重新拼装；
  - 基本面数据天然适合本地持久化与离线刷新。
- **详情**：
  - `POST /api/v1/valuecell` 主链路改为：**读 SQLite → 计算派生指标/warnings → 返回现有 JSON shape**
  - 在线 API 不再作为请求期主查询链路，只承担：
    - 批量初始化
    - 手动刷新
    - 周期性补数

### 2) v1 范围冻结：先服务美股核心字段
- **决策/推荐**：v1 只覆盖 ValueCell 当前最需要的**美股核心结构化财报数据 + latest quote metrics**。
- **理由**：
  - 现有 `FinancialMetrics`、前端页面、测试样本主要围绕美股链路；
  - 先把单市场单口径跑通，比一开始抽象成多市场统一模型更稳。
- **详情**：
  - 纳入 v1：
    - 公司主数据
    - 最近 10 年年度基本面
    - 最新行情/估值摘要
    - ETL 运行日志与问题表
  - 不纳入 v1：
    - 季报
    - 分部信息
    - 财报全文/PDF
    - 多市场统一币种换算框架

### 3) 数据库存储位置冻结
- **决策/推荐**：SQLite 文件固定为 `finguard/data/fundamentals.db`。
- **理由**：
  - 明确与当前零散缓存区分；
  - 便于后端服务按稳定相对路径读取；
  - 便于后续打包、备份、导出。
- **详情**：
  - `finguard/data/` 作为长期数据目录；
  - `uploads/tmp/*.json` 只视为过渡缓存，不再作为长期数据资产设计基础。

### 4) 表设计冻结：宽表 + 少量辅助表
- **决策/推荐**：采用“**宽表 + 少量辅助表**”模型，不做 EAV / fact 万能模型。
- **理由**：
  - ValueCell 当前查询模式稳定、字段集合明确；
  - 宽表更适合本地查询、调试、人工核对与导出；
  - 可以降低 ETL 与 C++ 读取层复杂度。
- **详情**：

#### 4.1 `companies`
- 主键：`symbol`
- 字段：
  - `symbol`
  - `normalized_symbol`
  - `company_name`
  - `market`
  - `currency`
  - `sector`
  - `industry`
  - `is_active`
  - `created_at`
  - `updated_at`

#### 4.2 `annual_fundamentals`
- 主键：`(symbol, fiscal_year)`
- 核心字段：
  - `revenue`
  - `operating_income`
  - `net_income`
  - `eps_diluted`
  - `shares_diluted`
  - `total_assets`
  - `total_liabilities`
  - `total_equity`
  - `operating_cash_flow`
  - `capex`
  - `free_cash_flow`
- 派生可落库字段：
  - `book_value_per_share`
  - `roe`
  - `debt_ratio`
- 元信息字段：
  - `source`
  - `source_updated_at`
  - `quality_flag`
  - `ingest_run_id`

#### 4.3 `latest_quote_metrics`
- 主键：`symbol`
- 字段：
  - `price`
  - `trailing_pe`
  - `forward_pe`
  - `price_to_book`
  - `peg_ratio`
  - `trailing_eps`
  - `market_cap`
- 元信息字段：
  - `source`
  - `quote_time`
  - `updated_at`

#### 4.4 `ingestion_runs`
- 主键：`id`
- 字段：
  - `run_type`（`bootstrap/manual_refresh/scheduled_refresh`）
  - `symbol_scope`
  - `status`
  - `started_at`
  - `finished_at`
  - `error_summary`

#### 4.5 `ingestion_issues`
- 字段：
  - `id`
  - `ingest_run_id`
  - `symbol`
  - `stage`
  - `issue_code`
  - `issue_message`
  - `created_at`

### 5) ETL 形态冻结：Python 入库，C++ 读库
- **决策/推荐**：v1 采用 **Python ETL 入库 + C++ 服务读库** 的职责分工。
- **理由**：
  - 外部数据抓取、清洗、批量 upsert 更适合脚本/数据工具链；
  - C++ 服务应优先承担读库、派生计算、接口稳定性，不再扩张为抓取器；
  - 与现有 ValueCell 运行时职责分离，便于排障与验收。
- **详情**：
  - Python 负责：
    - 按 symbol 初始化入库
    - 年报数据 upsert
    - latest quote 批量刷新
    - ETL run / issue 落库
  - C++ 负责：
    - symbol 归一
    - SQLite 查询
    - 派生指标重算
    - warnings 组装
    - `/api/v1/valuecell` 响应拼装

### 6) 现有 ValueCell 接口兼容边界冻结
- **决策/推荐**：保持 `POST /api/v1/valuecell` 响应 shape 基本兼容，前端尽量不需要大改。
- **理由**：
  - 当前前端已稳定消费既有字段；
  - 这轮核心是底层数据来源改造，不是接口协议重写。
- **详情**：
  - 保持现有主字段：
    - `symbol / requested_symbol`
    - `price / eps / pe / peg / price_to_book`
    - `avg_roe_5y / debt_ratio / growth_5y_cagr`
    - `net_profit_8y / net_profit_history / warnings`
  - `FinancialMetrics` 对外语义保持不变，但数据来源改为 SQLite repository。

### 7) 派生指标职责冻结
- **决策/推荐**：数据库保存原始/半结构化年报字段，ValueCell 请求期重算核心派生指标。
- **理由**：
  - 关键公式仍需要与当前业务逻辑、warnings 规则、测试保持同步；
  - 让数据库保留“可复核原材料”，避免把所有业务语义固化到 ETL。
- **详情**：
  - 落库为主：
    - 年度原始字段
    - `book_value_per_share / roe / debt_ratio` 等低争议派生
  - 请求期重算：
    - `np_recent3_avg`
    - `np_past3_avg`
    - `growth_5y_cagr`
    - `pe`
    - `peg`
    - `valuation_label`

### 8) warnings 与降级语义冻结
- **决策/推荐**：在保持现有 warnings 兼容的基础上，新增数据库相关 warnings。
- **理由**：
  - 需要把“数据库缺失/陈旧/不足”的状态显式暴露给前端和测试；
  - 不能因改成本地库就丢失可解释性。
- **详情**：
  - 继续保留：
    - `missing_debt_ratio`
    - `missing_roe_history`
    - `insufficient_net_profit_history`
    - `non_positive_growth`
    - `low_growth_5y_cagr`
  - 新增：
    - `db_symbol_not_found`
    - `db_insufficient_annual_history`
    - `db_missing_latest_quote`
    - `db_stale_quote_metrics`

### 9) Symbol 归一规则冻结
- **决策/推荐**：继续保留现有 alias 归一逻辑，尤其是 `BRK -> BRK-B` 兼容。
- **理由**：这是已验证过的用户输入兼容需求，不能在切库后回退。
- **详情**：
  - 请求入口先归一 symbol；
  - 数据库内部统一保存规范 symbol；
  - 响应继续保留 `requested_symbol` 便于核对。

### 10) 旧 JSON cache 处理策略冻结
- **决策/推荐**：不做旧 `uploads/tmp/*.json` 的正式迁移，v1 以后续 SQLite 重建为准。
- **理由**：
  - 旧缓存格式分散、时效性不稳定、不能代表长期 schema；
  - 直接迁移会把临时权宜结构固化为历史包袱。
- **详情**：
  - 旧缓存可在过渡期保留作为应急参考；
  - 文档与实现都不以旧 JSON cache 为真源；
  - 待 SQLite 方案跑稳后，再评估清理策略。

### 11) 前端验证入口冻结：SettingsPage 切库，ValueCellPage 验证
- **决策/推荐**：前端成品闭环固定为 **SettingsPage 选择数据库 profile → ValueCellPage 分析验证**，不新增独立验证页。
- **理由**：
  - 用户最终需要的是“直接在现有前端验证数据库表现”，而不是继续依赖环境变量或开发命令；
  - 现有 Settings / ValueCell 页面已经是最短产品路径，只需把“切库”产品化。
- **详情**：
  - 固定支持四个 profile：
    - `main` → `finguard/data/fundamentals.db`
    - `tcom_demo` → `finguard/data/examples/tcom_fundamentals_example.db`
    - `brk_b_demo` → `finguard/data/examples/brk-b_fundamentals_example.db`
    - `pdd_demo` → `finguard/data/examples/pdd_fundamentals_example.db`
  - 路径解析优先级固定为：
    1. `FINGUARD_FUNDAMENTALS_DB_PATH`
    2. `config/valuation.json` 中持久化的 `valuecell_db_profile`
    3. 默认 `main`
  - `GET /api/v1/settings` / `POST /api/v1/settings` 纳入 `valuecell_db_profile`
  - `POST /api/v1/valuecell` 响应新增可选元信息：
    - `db_profile`
    - `db_profile_label`
  - 默认不开放任意 SQLite 路径输入，避免把验证 UI 做成开发工具面板。

## Chapter 2：Database / ETL / ValueCell 实施计划

### 当前阶段目标
- 建立 ValueCell 的本地基本面数据库主方案；
- 用 SQLite 替换请求期在线主数据源；
- 把“环境变量切库原型”收敛成前端可直接验证的成品闭环；
- 为后续实现提供清晰、可执行、可验收的单一主文档。

### 实施步骤
1. **建立数据库基础设施**
   - 创建 `finguard/data/` 数据目录约定；
   - 设计 SQLite schema、索引、初始化脚本；
   - 明确版本化建库方式。
2. **建立 ETL 写库能力**
   - 支持按 symbol 初始化写入公司主数据；
   - 支持 10 年年度基本面 upsert；
   - 支持 latest quote metrics 刷新；
   - 记录 ingestion run 与 issue。
3. **建立后端读库层**
   - 新增 SQLite repository；
   - 按 symbol 读取公司、年度财务、最新行情；
   - 返回 ValueCell 所需最小数据集。
4. **接入 ValueCell 主链路**
   - `POST /api/v1/valuecell` 改为读库；
   - 保持 `FinancialMetrics` 外部语义与响应 shape 基本兼容；
   - 保留 alias 归一与 warnings 逻辑。
5. **补齐测试护栏**
   - schema/init 测试；
   - upsert 幂等测试；
   - repository 读取测试；
   - stale / missing / insufficient history 测试；
   - ValueCell 接口回归测试。
6. **完成人工样本验收**
   - 固定样本：`AAPL / BRK / PDD / TCOM`
   - 验收离线读库结果、warnings、前端兼容性。
7. **完成前端可验证产品化**
   - 后端引入 profile 注册表与元信息接口；
   - SettingsPage 支持持久化切换 DB profile；
   - ValueCellPage 显示当前验证来源，并在 demo 模式下提示样本 symbol；
   - 示例库生成与验收脚本统一收口。

### 推荐实施拆分
- **第一波**：schema + 初始化脚本 + ETL 骨架
- **第二波**：C++ repository + ValueCell 切库
- **第三波**：测试补强 + 样本数据验收 + 旧 cache 弃用整理
- **第四波**：前端可验证成品收口（profile 注册表 / Settings / ValueCell badge / demo DB 工具）

### 问题与处理策略

#### 问题 1：当前已经存在零散本地缓存，容易与新数据库职责重叠
- **现象**：`uploads/tmp/*.json` 已承担部分缓存职责。
- **处理策略**：
  - 明确旧 cache 不是长期真源；
  - 新方案直接围绕 SQLite 设计，不围绕 JSON cache 演化。

#### 问题 2：若请求期继续联网补数，会削弱“本地库主方案”的意义
- **现象**：如果数据库缺失就实时联网，系统仍会回到旧的不稳定模式。
- **处理策略**：
  - 请求期默认不联网；
  - 缺失时返回可解释 warnings；
  - 在线数据抓取只保留在离线 ETL。

#### 问题 3：切库后最容易出现前端字段兼容回归
- **现象**：底层来源变化后，字段缺失与 warnings 语义可能漂移。
- **处理策略**：
  - 保持 `FinancialMetrics` 对外语义稳定；
  - 先补 repository / warnings / alias 回归测试，再接主链路。

#### 问题 4：本机构建环境没有现成可直接链接的 SQLite 依赖
- **现象**：仓库现有 vcpkg 安装集中未直接暴露可立即复用的 SQLite 开发库，若继续等待环境补齐会阻塞本轮实施。
- **处理策略**：
  - 将本机已有 `sqlite3.c / sqlite3.h` 以 vendored 方式纳入 `finguard/third_party/sqlite/`
  - CMake 改为直接编译 vendored sqlite 源，避免外部依赖差异
- **结果**：Release 构建通过，SQLite 基础层顺利落地。

#### 问题 5：单元测试污染了正式验收数据库
- **现象**：初版测试直接使用 `finguard/data/fundamentals.db`，导致测试执行后覆盖掉刚 seed 的正式验收数据。
- **处理策略**：
  - 在 `fundamentals_db.cpp` 中新增环境变量覆盖：`FINGUARD_FUNDAMENTALS_DB_PATH`
  - 测试改为使用独立临时库 `test_fundamentals.db`
- **结果**：测试与验收数据完全隔离，避免互相污染。

#### 问题 6：本地离线样本只有 8 年，尚未满足“10 年数据可读”理想目标
- **现象**：当前仓库可复用的样本/缓存主要覆盖最近 8 年，不足以完整验证 10 年历史。
- **处理策略**：
  - 先保证 schema、ETL、读库链路、warnings、接口兼容全部落地
  - 在 `ingestion_issues` 与验收日志中明确记录该数据覆盖缺口
- **结果**：核心功能实现完成，但“10 年样本覆盖”验收项保留为部分通过。

#### 问题 7：本机环境变量会污染 profile resolver 测试
- **现象**：在本机已有 `FINGUARD_FUNDAMENTALS_DB_PATH` 时，`BuiltinProfilesResolveExpectedKeys` 会错误解析到 `env_override`，导致测试不稳定。
- **处理策略**：
  - 在测试中新增 `EnvVarGuard`
  - 对“内建 profile 解析”用例显式清空环境变量，并在测试结束后恢复
- **结果**：`finguard_tests.exe` 重新稳定为 **53/53 通过**。

#### 问题 8：Settings 持久化需要保证 LLM 配置与 Valuation 配置分文件写入
- **现象**：若直接沿用旧写法，容易把 `valuecell_db_profile` 混写到 `llm.json`，或把空 JSON 文档写成 `null`。
- **处理策略**：
  - `system_routes.cpp` 中分别维护 `llm_doc` 与 `valuation_doc`
  - 默认初始化为 `nlohmann::json::object()`
  - `valuecell_db_profile` 只写入 `config/valuation.json`
- **结果**：保存设置后立即生效，且未破坏原有 LLM 设置文件结构。

#### 问题 9：投研问答与基本面分析曾分裂成两套 prompt 体系
- **现象**：
  - Chat 原先把用户原始 prompt 直接传给 LLM；
  - ValueCell 历史版本则围绕搜索摘要构造定性 prompt；
  - 两者都没有稳定把 SQLite / 结构化财务数据作为统一上下文。
- **处理策略**：
  - 抽出统一 `analysis_workflow`
  - 先完成 symbol 提取、DB 读取、自动联网补数、可选写回 DB
  - 再将统一结构化上下文与用户问题一起交给 LLM
- **结果**：当前 Chat 与 ValueCell 已统一为“结构化财务数据优先”的单一路线。

#### 问题 10：数据库没有该公司时，旧行为只能报缺失，不能自动补数
- **现象**：用户输入 DB 中没有的公司时，ValueCell 只能返回缺失告警，无法顺手补进主库。
- **处理策略**：
  - 为 SQLite 基础层新增 upsert helper：
    - `upsert_company_record(...)`
    - `upsert_annual_fundamentals_records(...)`
    - `upsert_latest_quote_metrics_record(...)`
  - 当 DB miss 或结构化数据不完整时，优先尝试现有结构化抓取链路
  - 在 `main` profile 下成功抓取后自动回写 SQLite 主库
- **结果**：`MSFT` 等主库缺失 symbol 已可触发“联网抓取 → 本次分析使用 → 尝试写回主库”闭环。
