# Phase Asset-Allocation-Multi-Rules — 资产配置多规则框架建设

## Chapter 1：Spec 前置计划（冻结）

本阶段目标：在现有守拙理念规则基础上，集成桥水全天候、永久投资组合、斯文森耶鲁三个经典规则，让用户能够对比选择。

### 1) 四大规则定义与标准配置

**决策/推荐**：采用国际资产版本，支持全球券商（美股+港股+A股）

#### 规则1：守拙价值多元化基金理念（Shouzhe Value Diversification Fund Concept）
- **资产配置**：
  - 黄金 (GLD): 10% (固定)
  - 债券 (BND): 年龄相关 = (age/10)*10%
  - 股票: 余量 (age-dependent)
    - 美股 (VOO): 港A美=1:3:16中的美股部分
    - A股 (HS300/CSI300): 港A美=1:3:16中的A股部分
    - 港股 (HSI): 港A美=1:3:16中的港股部分
- **个股权重**：支持投资人分层 (novice/experienced/professional)
- **特点**：年龄驱动，个股可选，最贴近国内用户

#### 规则2：桥水全天候策略（Bridgewater All-Weather Strategy）
- **资产配置**（固定，所有年龄相同）：
  - 股票 (VOO): 30%
  - 长期债券 (BND/TLT): 40%
  - TIPS(通膨保护债券): 15%
  - 大宗商品/黄金 (GSG/GLD): 15%
- **特点**：风险平衡，固定权重，极简，适合机构投资者
- **预期收益**：8-12% | **最大回撤**：-10~-15%

#### 规则3：永久投资组合（Permanent Portfolio）
- **资产配置**（固定，所有年龄相同，完美平衡）：
  - 股票 (VOO): 25%
  - 长期债券 (TLT): 25%
  - 黄金 (GLD): 25%
  - 现金/货币基金 (SHV): 25%
- **特点**：最简单配置，4资产等权重，懒人投资首选
- **预期收益**：6-9% | **最大回撤**：-8~-12%

#### 规则4：斯文森耶鲁配置规则（Swensen Yale Model）
- **资产配置**（固定，所有年龄相同，多元化）：
  - 美国股票 (VOO/VTI): 30%
  - 国际发达市场 (VEA/IXUS): 15%
  - 新兴市场 (VWO/IEMG): 10%
  - 房产/REITs (VNQ/XLRE): 20%
  - 债券 (BND/AGG): 15%
  - 黄金 (GLD): 10%
- **特点**：6资产高度多元化，学术背书（美国名校捐赠基金），追求长期α收益
- **预期收益**：10-15% | **最大回撤**：-15~-25%

### 2) 前端交互模式

**决策/推荐**：多选对比模式 (方案A)

- **交互流程**：
  1. 用户在前端看到4个规则的卡片（复选框）
  2. 用户勾选想要对比的规则组合（默认勾选"守拙理念"）
  3. 点击"生成对比方案"按钮
  4. 前端展示对比表格：资产分配、风险指标、预期收益、适用人群等
  5. 用户可选择其中一个规则，确认应用

- **推荐展示优化**：
  - 基于用户画像自动推荐首选规则（带✨标记）
  - 显示规则间的核心差异（股票占比、风险水平等）
  - 提供"详细说明"链接，指向规则的白话解释

**不采用单选切换**：避免用户漏掉对比的价值

### 3) API 返回结构

**决策/推荐**：单规则模式 (方案A)

- **API 设计**：
  ```
  POST /api/v1/plan?rule=<rule_name>
  
  rule 参数可选值：
    - "shouzhe"      (默认，守拙理念)
    - "bridgewater"  (桥水全天候)
    - "permanent"    (永久投资组合)
    - "swensen"      (斯文森耶鲁)
  
  请求体保持不变：
  {
    "profile": { age, investor_type, experience_years, ... },
    "portfolio": [ ... ]  // 当前持仓 (可选)
  }
  ```

- **响应格式**：每次单规则返回
  ```json
  {
    "rule_name": "shouzhe",
    "proposed_portfolio": {
      "VOO": 0.384,
      "HSI": 0.038,
      "HS300": 0.115,
      "BND": 0.30,
      "GLD": 0.10,
      "individual_stock": 0.073
    },
    "risk_report": { ... },
    "rationale": "...",
    "rebalancing_actions": [ ... ]
  }
  ```

- **前端对比**：前端负责多次调用 `/api/v1/plan?rule=xxx`，收集结果后展示对比表

**理由**：
- 后端逻辑清晰，单一职责
- 前端全权管理对比逻辑
- 便于缓存与性能优化

### 5) 投资人分层策略

**决策/推荐**：只对守拙理念应用分层，其他三个规则均为固定配置

**理由**：
- 其他三个规则已是学术验证的经典方案，分层会破坏其纯正性
- 简化实现复杂度
- 用户可通过选择不同规则来实现"分层"效果

### 问题与解决

- 问题 1：如何在支持多规则的同时保持 `routes.cpp` 简洁、避免“胖路由”。
  - 定位：原 `/api/v1/plan` 路由中同时包含参数解析、规则计算、风控、调仓建议，耦合度高。
  - 处理：新增 `src/server/allocation_handler.h/cpp`，将业务逻辑下沉到 handler 层，路由仅保留请求校验和调用编排。

- 问题 2：如何在新增三套规则时保持代码结构可维护。
  - 定位：若继续在 `routes.cpp` 追加逻辑，会违反单一职责并增加后续维护成本。
  - 处理：按模块拆分 `src/core/allocation_rule.*`、`src/core/rules/*`、`src/core/rule_factory.*`，遵循“一文件一职责”。

- 问题 3：如何保证新增规则不破坏现有接口行为。
  - 定位：接口扩展有兼容性风险。
  - 处理：沿用原 `POST /api/v1/plan`，仅增加可选查询参数 `rule`，默认值保持 `shouzhe`。

- 问题 4：联调环境对 HTTP 调试命令存在策略限制。
  - 定位：当前执行策略禁止 `Invoke-RestMethod`、`curl`，无法直接做命令行实时探测。
  - 处理：改用替代验收路径（后端编译 + 单测全通过 + 前端构建通过），并在 Chapter 4 明确记录限制与后续建议。
